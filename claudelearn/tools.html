エージェントハーネスの正体を見に行く

前のセクションで「エージェントハーネス」という言葉が出てきました。 ここからは実際にその正体を確認していきましょう。

半分当たってた話

Claude CodeのGitHubリポジトリ（anthropics/claude-code）を見ると、 TypeScriptのソースコードが並んでいます。「これがClaude Codeの本体か」と思いたくなりますが、 実はこれはプラグイン集（ツール定義やHooks等）であり、本体ではありません。

インストール先を確認する

Claude Codeは curl -fsSL https://claude.ai/install.sh | bash でインストールされます （npmインストールは非推奨）。 実際にどこに何がインストールされているか確認してみましょう。

which claude
/Users/you/.local/bin/claude
ls -la ~/.local/share/claude/
claude-code-{version}/ ← ネイティブバイナリ本体

つまり

claude コマンドの実体は、たった1つのファイル cli.js です。このファイルが約10MBあり、 ミニファイ（圧縮）された状態で全てのロジックがバンドルされています。

cli.js の中身をのぞく

ミニファイされているので人間には読みにくいですが、キーワード検索はできます。 実際に検索してみると、Claude Codeの仕組みが透けて見えます。

grep -c "toolUse\|tool_use" cli.js

grep -c "systemPrompt\|system_prompt" cli.js

grep -c "permission\|Permission" cli.js

なぜこれが重要か

GitHubリポジトリを読んでも「エージェントループの核心部分」は見えません。 それはcli.jsにバンドルされているからです。しかし、ツールの定義方法や 拡張の仕組みはリポジトリから学べます。つまり「半分当たってた」のです。

ツール定義の実態 ー JSONスキーマ

LLMにとってツールとは何でしょうか？ 答えは「テキストで説明された関数のインターフェース」です。 実際のツール定義を見てみましょう。

ツール定義の3要素

全てのツールは、以下の3つの要素で定義されています。

// Bashツールの定義（簡略版）
{
"name": "Bash",
"description": "コマンドを実行する。
catやgrepではなくRead、Grepツールを使え。
sedやawkではなくEditツールを使え...",
"input_schema": {
"type": "object",
"properties": {
"command": {
"type": "string",
"description": "実行するコマンド"
},
"timeout": {
"type": "number",
"description": "タイムアウト（ミリ秒）"
}
},
"required": ["command"]
}
}
name

ツールの識別名。LLMが tool_use レスポンスで指定する名前。

description

LLMへの説明文。いつ・どう使うべきかの戦略的指示を含む。

input_schema

パラメータの型定義。JSON Schemaで引数の形を規定する。

descriptionの戦略性

ツールの description は単なる説明ではありません。 「catではなくReadツールを使え」「grepコマンドではなくGrepツールを使え」といった 戦略的な指示がここに含まれています。 LLMはこの説明を読んで、どのツールを選ぶか判断します。

LLMにとっての「ツール」

LLMはツールの実装コードを一切見ていません。 name・description・input_schema という「テキスト情報」だけを見て、 どのツールをどう使うか判断しています。 だからこそ、descriptionの書き方がClaude Codeの品質に直結するのです。

ツール実行フロー ー tool_use から tool_result まで

LLMがツールを使いたいと言ってから、結果が返るまでの詳細なフローを見てみましょう。

ツール実行の詳細フロー
Claude APIが tool_use を返す
「Bashツールで ls -la を実行して」
↓
パーミッション確認
このツール呼び出しは許可されているか？
許可
→ 実行へ進む
拒否
→ エラーを返す
↓
ツールをローカルPCで実行

↓

実際のデータで見る

// 1. LLMの応答（tool_use）
{
"type": "tool_use",
"id": "toolu_01ABC...",
"name": "Bash",
"input": {
"command": "ls -la",
"description": "List files in current directory"
}
}
// 2. エージェントハーネスが実行後、結果を返す（tool_result）
{
"type": "tool_result",
"tool_use_id": "toolu_01ABC...",
"content": "total 48\ndrwxr-xr-x 5 user staff ..."
}

tool_use_id による紐づけ

tool_use_id は、LLMのツール呼び出しと実行結果を紐づけるIDです。 LLMは1回の応答で複数のツールを呼び出すことがあり、 このIDによってどの結果がどの呼び出しに対応するか判別します。

各ツールの詳細と使い分け

Claude Codeが使える主要なツールを、それぞれの役割と使い分けの観点から見ていきましょう。

Bash
シェルコマンドを実行する万能ツール

任意のシェルコマンドを実行できる最も強力なツール。 ただし、システムプロンプトで「ファイル読み込みには Read を使え」 「検索には Grep を使え」と制限されています。

主な用途： npm install、 git 操作、 docker コマンド、 ビルド・テスト実行など

Read
ファイルの内容を読み取る専用ツール

ファイルを読むための専用ツール。cat や head の代わりに使用されます。 行番号付きで表示され、画像やPDFも読めます。

なぜcat ではダメなのか： 専用ツールを使うことで、ユーザーが「何のファイルを読んだか」を 明確に把握でき、パーミッション制御も効きます。

Edit / Write
ファイルの編集・作成ツール

Edit は既存ファイルの部分的な変更（差分ベース）、 Write は新規ファイルの作成または全体の書き換え。 sed や awk の代わりに使用されます。

Editの仕組み： old_string（変更前）と new_string（変更後）を指定する文字列置換方式。 差分が明確なので、ユーザーが変更を確認しやすい。

Grep / Glob
コードベースの検索ツール

Grep はファイル内容の正規表現検索（ripgrep ベース）、 Glob はファイル名のパターンマッチ。 Bashで grep や find を使うことは禁止されています。

専用ツールの利点： パーミッション最適化、出力フォーマット統一、パフォーマンス向上。

Agent
サブエージェントを起動するツール

独立したコンテキストウィンドウを持つサブエージェントを起動します。 複雑なタスクの並列処理や、メインのコンテキストウィンドウを 汚さないための調査作業に使用されます。

特徴： サブエージェントは独自のシステムプロンプトとツールセットを持つ。 結果だけがメインエージェントに返される。

WebFetch / WebSearch
Webからの情報取得ツール

WebFetch は指定URLのページ取得、 WebSearch はWeb検索。 ドキュメント参照やAPI仕様の確認に使われます。

「Bashがあれば全部できる」のに、なぜ専用ツールがあるのか？

理由は3つあります：

  • 安全性 — 専用ツールは操作範囲が限定されている。Bashは何でもできてしまう
  • 可視性 — 「Readツールでファイルを読んだ」はユーザーに明確。「Bashでcatした」は意図が不明瞭
  • パーミッション制御 — ツールごとに許可/拒否の粒度で制御できる

パーミッションシステム ー なぜ確認が出るのか

Claude Codeを使っていると「このツールの実行を許可しますか？」という確認が出ます。 これはパーミッションシステムが動いているからです。

パーミッションの仕組み

ツール呼び出しの度に、エージェントハーネスが以下のチェックを行います。

1
ツールの種類を確認
Read は安全、Bash は危険かもしれない、Edit はファイルを変更する
2
パーミッション設定と照合
ユーザーが事前に許可/拒否しているかチェック
3
必要に応じてユーザーに確認
未設定の操作は、実行前にユーザーの承認を求める

パーミッションモード

Plan モード

読み取りのみ。ファイル変更やコマンド実行は不可。調査・計画に使用。

Default モード

安全な操作は自動許可。ファイル変更やBashコマンドは確認を求める。

Yolo モード

ほぼ全ての操作を自動許可。高速だが注意が必要。

パーミッション ＝ エージェントハーネスの仕事

パーミッション確認はLLM側では一切行われていません。 エージェントハーネスが tool_use を受け取った時点で、 ローカルで判断しています。LLMが「許可が必要」と言っているわけではなく、 ハーネスが「この操作は確認が必要」と判定しているのです。

サブエージェントの仕組み

Claude Codeには「エージェントの中にエージェントを起動する」機能があります。 これがサブエージェントです。

サブエージェントの構造
メインエージェント
コンテキストウィンドウ
ユーザーとの全会話履歴・システムプロンプト・全ツール定義
↓ Agent ツールを呼び出す
サブエージェント A
- 独自のコンテキストウィンドウ
- 独自のシステムプロンプト
- 限定されたツールセット
例: コードベース探索
サブエージェント B
- 独自のコンテキストウィンドウ
- 独自のシステムプロンプト
- 限定されたツールセット
例: テスト実行
↑ 結果だけがメインに返される

サブエージェントのメリット

コンテキスト分離

大量の検索結果や調査内容がメインのコンテキストウィンドウを圧迫しない。 結果のサマリーだけが返される。

並列処理

複数のサブエージェントを同時に起動し、独立したタスクを並列実行できる。

専門化

サブエージェントごとに異なるシステムプロンプトとツールセットを持てる。 探索用、テスト用など用途に特化できる。

コスト効率

メインの会話履歴を送らないため、トークン消費を抑えられる場合がある。

サブエージェント = 別のエージェントループ

サブエージェントは、メインエージェントとは独立した エージェントループを実行します。自分自身のシステムプロンプト、 ツール定義、会話履歴を持ち、タスクが完了すると結果だけをメインに返します。 「エージェントの中のエージェント」ですが、仕組みは全く同じです。

まとめと次のステップ

このセクションで学んだこと

  • Claude Codeの本体は cli.js（約10MB）にバンドルされている
  • GitHubリポジトリはプラグイン集であり、コアのエージェントループは含まれない
  • ツールはJSONスキーマ（name, description, input_schema）で定義される
  • description にはLLMへの戦略的な指示が含まれている
  • ツール実行フローは tool_use → パーミッション確認 → 実行 → tool_result
  • 専用ツール（Read, Grep等）がBashの代わりに使われる理由は安全性・可視性・制御性
  • パーミッション確認はエージェントハーネス側で行われる
  • サブエージェントは独立したエージェントループを持つ
