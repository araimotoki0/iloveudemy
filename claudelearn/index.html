Claude Codeの正体 ー 「賢いAI」ではなく「仕組み」

Claude Codeを使うと、AIがファイルを読み、コードを書き、コマンドを実行してくれます。 まるでAIが「考えて行動している」ように見えますが、実際には Claude（LLM）自体はテキストを受け取り、テキストを返すことしかできません。

よくある誤解

「Claude Codeは超高性能なAIだから、ファイル操作やコマンド実行ができる」

実際の仕組み

Claude（LLM）は「このツールを使いたい」という意図をテキストで返すだけ。 実際にファイルを読み書きするのは、エージェントハーネス（CLIプログラム）の仕事です。

2つのコンポーネント

サーバー側

Claude API（LLM）

Anthropicのサーバーで動作。テキストを入力すると、テキスト（+ ツール呼び出し意図）を返す。ツールの実行はできない。

クライアント側

エージェントハーネス（CLI）

あなたのPCで動作。APIとの通信、ツールの実行、結果のフィードバックを行うループプログラム。

この講座で伝えたいこと

Claude Codeの「すごさ」は、LLMの能力だけでなく、エージェントハーネスの設計にあります。 この仕組みを理解すれば、使い方が根本的に変わります。

エージェントループ ー Claude Codeの心臓部

Claude Codeの全ての動作は、たった1つのループで説明できます。 これがエージェントループです。

エージェントループの全体像
1
メッセージ送信
ユーザーの入力 + システムプロンプト + 会話履歴を Claude API に送信
↓
2
応答を受信
Claude APIがテキスト応答を返す。応答には2種類ある：
A) テキストのみ
→ ユーザーに表示して終了
B) ツール呼び出し
→ ステップ3へ
↓
3
ツール実行
エージェントハーネスがツールをローカルPC上で実行する
↓
4
結果をAPIに送り返す
ツールの実行結果を会話に追加し、ステップ1に戻る
↩ ツール呼び出しがなくなるまでループ

最重要ポイント

Claude（LLM）は「Bashツールを使って ls -la を実行してください」というテキストを返すだけです。 実際に ls -la を実行するのはエージェントハーネス。 LLMは一切コマンドを実行していません。

API通信の実態 ー 何が送受信されているか

エージェントループで実際にやりとりされるデータを見てみましょう。

1. ユーザーが「このディレクトリのファイルを見せて」と入力

// エージェントハーネス → Claude API（リクエスト）
{
"model": "claude-sonnet-4-20250514",
"system": "あなたはClaude Codeです。Bashツール、Readツール...",
"tools": [
{ "name": "Bash", "description": "コマンドを実行する" },
{ "name": "Read", "description": "ファイルを読む" }, ...
],
"messages": [
{ "role": "user", "content": "このディレクトリのファイルを見せて" }
]
}
// Claude API → エージェントハーネス（レスポンス）
{
"content": [{
"type": "tool_use",
"name": "Bash",
"input": { "command": "ls -la" }
}],
"stop_reason": "tool_use"
}

注目ポイント

  • stop_reason: "tool_use" — LLMは「まだ終わっていない、ツールの結果を待っている」
  • tools 配列 — 使えるツールの定義はエージェントハーネスが毎回送っている
  • LLMは ls -la を実行していない。「実行してほしい」と返しただけ

2. ツール実行後、結果を送り返す

// エージェントハーネス → Claude API（ツール結果を含む2回目のリクエスト）
"messages": [
{ "role": "user", "content": "このディレクトリのファイルを見せて" },
{ "role": "assistant", "content": [{ "type": "tool_use", ... }] },
{ "role": "user", "content": [{
"type": "tool_result",
"content": "total 48\ndrwxr-xr-x 5 user staff ..."
}] }
]
// Claude API → エージェントハーネス（最終レスポンス）
{
"content": [{
"type": "text",
"text": "このディレクトリには以下のファイルがあります..."
}],
"stop_reason": "end_turn"
}

ループ完了

stop_reason: "end_turn" でループ終了。 今回はAPIを2往復しました。複雑なタスクでは数十往復することもあります。

ツールの仕組み ー LLMの「手足」

Claude Codeが使えるツールは全てエージェントハーネス側で定義されています。

主要なツール一覧

Bash

シェルコマンドを実行

Read

ファイルの内容を読み取る

Edit / Write

ファイルの編集・作成

Grep / Glob

コードベースの検索

Agent

サブエージェントを起動

WebFetch

Webページの取得

ツールの本質

ツールはJSONスキーマで定義されています。 LLMにとってツールは「テキストで説明された関数のインターフェース」に過ぎません。 実装はエージェントハーネス側にあります。

システムプロンプト ー Claudeへの「取扱説明書」

エージェントハーネスが毎回送信する巨大なシステムプロンプトが、Claude Codeの振る舞いを規定しています。

システムプロンプトに含まれるもの

1
ツールの定義と使い方
「Bashツールはこう使え」「Readツールでファイルを読め」
2
行動規範
「危険な操作は確認を取れ」「セキュリティ脆弱性を作るな」
3
環境情報
OS、シェル、カレントディレクトリ、gitブランチなど
4
CLAUDE.md の内容
ユーザーが定義したプロジェクト固有の指示

なぜこれが重要か

LLM自体は「自分がClaude Codeである」ことすら知りません。 システムプロンプトで教えているからこそ、ソフトウェアエンジニアのように振る舞えるのです。 プロンプトを変えれば、同じLLMで全く違うエージェントを作れます。

全体アーキテクチャ ー 全てを繋げる

Claude Code アーキテクチャ全体像
あなたのPC（クライアント側）
ユーザー
プロンプト入力
エージェントハーネス
ループ制御・ツール実行
ツール群
Bash, Read, Edit, Grep...
↕ HTTP（Messages API）
Anthropic サーバー
Claude（LLM）
テキスト入力 → テキスト + ツール呼び出し意図 を出力
※ ファイル操作やコマンド実行は一切できない

この図が理解できれば

  • なぜネットワークが必要か — LLMはリモートサーバーにいるから
  • なぜパーミッション確認があるか — ツール実行はローカルPCで行われるから
  • なぜCLAUDE.mdが効くか — システムプロンプトとしてAPIに毎回送られるから
  • なぜトークン消費が多いか — ループの度に全会話履歴を送るから

まとめと次のステップ

このセクションで学んだこと

  • Claude Codeは「エージェントループ」という仕組みで動いている
  • LLMはテキストを返すだけ。ツールを実行するのはエージェントハーネス
  • ツールはJSONスキーマで定義され、LLMには説明文として渡される
  • システムプロンプトがClaude Codeの「人格」と「能力」を規定している
