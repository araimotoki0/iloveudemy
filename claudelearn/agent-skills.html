Agent Skills とは何か

AI への依頼をファイルにして保存しておく仕組みです。
必要になった時に AI が自動でそのファイルを読み、書かれた手順どおりに作業します。
スキルにすると何度でも再利用できるから、繰り返し作業があればスキルにすると効率が良くなる

たとえば

  • 「PDF を要約する時はこの観点で見て、この形式で出力してね」
  • 「リリースノートを書くときは、こういう順序で項目を並べて」
  • 「CSV を渡されたらこのスクリプトで前処理してから分析して」

こういう「いつもやってもらう手順」を Skill としてファイルに残しておけば、毎回プロンプトに書かなくて済みます。

プロンプトのテンプレ集とも違うのは、AI が「いつ使うか」を自分で判断する点です。 ファイルには「どんな時に使う Skill か」を書いておきます。AI はその説明を見て、関連する依頼が来た時に自動で呼び出します。

正体はただのフォルダとファイル

Skill は1 つのフォルダです。中に SKILL.md という Markdown ファイルがあれば、それで Skill になります。 必要に応じて画像、CSV、スクリプトなど他のファイルも一緒に入れられます。

最小構成

my-skill/
└── SKILL.md

スクリプトや資料を同梱した例

csv-analyzer/
├── SKILL.md # 手順書（必須）
├── scripts/
│ └── clean.py # AI が呼び出す前処理スクリプト
├── references/
│ └── format.md # 出力フォーマットの定義
└── assets/
└── template.xlsx # 雛形ファイル

AI は SKILL.md を読んで、必要なら scripts/clean.py を実行したり、template.xlsx を雛形として使ったりします。 「.md だけ」という縛りがないのがポイントです。手順 + 道具をまるごと配れます。

どこに置くか（配置場所）

Skill フォルダは 必ず skills/ ディレクトリの直下に置きます。 Claude Code はこの場所しかスキャンしないため、別の場所（例: .claude/my-skill/）に置いても認識されません。

スコープ 配置場所 使い分け
プロジェクト <repo>/.claude/skills/<skill名>/SKILL.md そのリポジトリでだけ使う Skill。チームで共有したい場合（git で commit）。
ユーザ全体 ~/.claude/skills/<skill名>/SKILL.md どのプロジェクトでも使う個人用 Skill。自分の PC 全体で有効。

配置後に Claude Code を再起動すると、/<skill名> で呼び出せるようになります。 認識されているかは、起動直後に読み込まれるスキル一覧に名前が出ているかで確認できます。

フォルダ名 = /コマンド名

フォルダ名はそのまま Claude Code のスラッシュコマンド名になります。 例えば ~/.claude/skills/deploy/ を作ると、Claude Code で /deploy と打って呼び出せるようになります。

フォルダ名（= skill 名）の命名ルール

公式仕様(agentskills.io)で決まっています。

  • 1〜64 文字
  • 小文字英数字(a-z 0-9)とハイフン(-)のみ
  • ハイフンで始まる / 終わる ❌
  • 連続ハイフン(--) ❌
例 可否
pdf-processing ✅ OK
code-review ✅ OK
PDF-Processing ❌ 大文字 NG
-pdf ❌ ハイフン始まり NG
pdf--processing ❌ 連続ハイフン NG

YAML の name はディレクトリ名と一致

SKILL.md の YAML ヘッダーにも name を書きますが、 公式仕様では「フォルダ名と一致しなければならない」と決まっています。 つまり実質、フォルダ名が真の identifier。Claude Code では name を省略するとフォルダ名が自動で使われるので、書かなくても OK です。

ハンズオン ── Python と Excel を同梱した実動 Skill

さっき「フォルダの中にスクリプトや雛形を一緒に入れられる」と説明しました。それを実際に動かしてみます。
作るのは 「売上 Excel を渡したら、しきい値を超えた行に色をつけて返してくれる Skill」。SKILL.md + Python スクリプト + サンプル xlsx の 3 点セットです。

1. フォルダ構成

~/.claude/skills/excel-highlight/ ├── SKILL.md ├── scripts/ │ └── colorize.py # openpyxl で行を塗るスクリプト └── assets/ └── sample.xlsx # 動作確認用のサンプルデータ

2. 事前準備（一度だけ）

Python と openpyxl を入れておきます。サンプル xlsx も Python で作ってしまうのが楽です。

mkdir -p ~/.claude/skills/excel-highlight/scripts ~/.claude/skills/excel-highlight/assets pip install openpyxl # サンプル xlsx を生成（売上データ 5 行） python3 - <<'PY' from openpyxl import Workbook wb = Workbook(); ws = wb.active; ws.title = "sales" ws.append(["商品", "売上"]) for name, v in [("りんご",80000),("みかん",120000),("ぶどう",95000),("いちご",210000),("もも",60000)]: ws.append([name, v]) wb.save("/Users/$USER/.claude/skills/excel-highlight/assets/sample.xlsx") PY

3. SKILL.md を書く

--- name: excel-highlight description: 売上 Excel を渡された時に使う。しきい値を超えた行を黄色で塗って別名保存する --- # Excel ハイライト手順 1. ユーザから渡された xlsx のパスと、しきい値（円）を確認する - しきい値が指定されなければ 100000 をデフォルトにする 2. `scripts/colorize.py <入力xlsx> <しきい値> <出力xlsx>` を実行する 3. 出力先は入力ファイルと同じディレクトリに `元のファイル名_highlighted.xlsx` で保存 4. 何行を塗ったかをユーザに日本語で報告する # 動作確認 サンプルが `assets/sample.xlsx` にある。動作確認を頼まれたらこれを使う。

4. scripts/colorize.py を書く

import sys from openpyxl import load_workbook from openpyxl.styles import PatternFill src, threshold, dst = sys.argv[1], int(sys.argv[2]), sys.argv[3] wb = load_workbook(src) ws = wb.active yellow = PatternFill("solid", fgColor="FFEB3B") painted = 0 for row in ws.iter_rows(min_row=2): # 1 行目はヘッダー value = row[1].value # B 列が売上 if isinstance(value, (int, float)) and value >= threshold: for cell in row: cell.fill = yellow painted += 1 wb.save(dst) print(f"painted={painted} threshold={threshold} out={dst}")

5. Claude Code で動かす

Claude Code を再起動してから、こう頼むだけです。「excel-highlight 使って」と指定する必要はありません。

~/.claude/skills/excel-highlight/assets/sample.xlsx の売上 10 万円以上の行に色をつけて

Claude は description（「売上 Excel を渡された時に使う」）からこの Skill を選び、SKILL.md の手順に従って colorize.py を呼びます。 結果は sample_highlighted.xlsx として同じフォルダに保存され、「3 行塗りました（みかん / いちご）」のように報告されます。

ここで体感してほしいこと

  • Skill = フォルダ。SKILL.md(指示書) + Python(道具) + xlsx(雛形) をまとめて配れる
  • AI に毎回「openpyxl で 2 列目を読んで PatternFill で…」と説明しなくていい。手順を一度書けば再利用できる
  • Python を呼ぶ部分は 普通のスクリプト。デバッグも単独で出来る（python3 scripts/colorize.py ... と直接叩けば良い）

SKILL.md の中身 ── YAML ヘッダー + 本文

ファイルの上に YAML 形式のヘッダーを書き、その下に Markdown で本文を書きます。

--- name: pdf-summary description: PDF を要約する時に使う。日本語で 5 行以内、各行に必ず数値を含めること --- # PDF 要約手順 1. ページ全体をざっと読み、テーマを把握する 2. 数字（金額、割合、件数など）を必ず拾う 3. 5 行以内で要約する 4. 各行に最低 1 つ数値を含める

description が一番大事

AI は最初にこの description だけを見て、「今の依頼にこの Skill を使うべきか」を判断します。 あいまいに書くと使われない、書きすぎるとコンテキストを食う。「いつ使うか」が一行で分かるように書くのがコツです。

ハンズオン ── 一番シンプルな Skill を作る

実際に作って動かしてみます。「コミットメッセージを規約に沿って書く」 Skill です。

1. フォルダを作る

mkdir -p ~/.claude/skills/commit-message

2. SKILL.md を書く

--- name: commit-message description: git commit メッセージを書くときに使う。Conventional Commits 形式 --- # コミットメッセージのルール - 形式: `<type>: <subject>` - type は feat / fix / docs / refactor / test / chore のいずれか - subject は 50 文字以内、現在形、文末にピリオドなし - 日本語可

3. Claude Code で使ってみる

Claude Code を起動して、こう聞くだけです。

この差分のコミットメッセージを考えて

description に「git commit メッセージを書くとき」と書いたので、Claude が自動で commit-message Skill を読み込み、 ルールどおりのメッセージを返してくれます。「Skill を使って」と指示する必要はありません。

応用 ── スクリプトを同梱する

フォルダの中に scripts/check.sh を置き、 SKILL.md に「メッセージを決めたら scripts/check.sh で文字数チェック」と書いておけば、 AI はそのスクリプトを実行して検証してくれます。手順 + 検証ツールをセットで配れるのが Skills の強みです。

なぜ Skills が効くのか ── コンテキストの問題

AI には「コンテキスト」（今この瞬間に読み込んでいる情報）の上限があります。 そして上限に近づくほど、古い情報を忘れたり、指示を取り違えたりして精度が落ちます。

「全部のルールを毎回読ませる」のがダメな理由

たとえばコーディング規約、コミット規約、レビュー観点、デザインルール…全部を CLAUDE.md に書いて毎回読ませると、 今の作業に関係ない情報まで AI の頭に詰め込まれます。結果、本当に集中してほしい部分の精度が下がるのです。

Skills が解く方法 ── 必要な時だけ読む

Skills は 3 段階でロードされます（Progressive Disclosure と呼ばれます）。

  1. 起動時: 全 Skill の name と description だけを読み込む（1 行 × N 個）
  2. マッチ時: 依頼に関連しそうな Skill だけ、本文を全部読み込む
  3. 実行時: 必要なら同梱スクリプトやファイルを呼び出す

100 個の Skill を持っていても、起動時に消費するコンテキストは「100 行のリスト」だけ。 関係ないルールでコンテキストを汚さない仕組みになっています。

配置場所と適用範囲

どこに置くかで「誰が使えるか」が決まります。

配置場所 適用範囲 使いどころ
~/.claude/skills/ 自分のマシン全体 どのプロジェクトでも使う個人用 Skill
.claude/skills/ そのプロジェクトだけ チームで Git に commit して共有

Claude は .claude/、業界は .agent/ 寄り

多くの AI ツールはプロジェクト直下の AGENTS.md や .agent/ ディレクトリを共通の置き場にしようという動きがあります。 一方 Claude Code は独自の .claude/ を使います。

ただしファイルの中身（SKILL.md 形式）は agentskills.io の標準に揃っているので、 フォルダの場所さえ違っても、書いたものは他ツールに移植できます。

agentskills.io ── オープン標準とその背景

agentskills.io は、Skill の書き方を統一するためのオープン標準です。 Anthropic が発案し、後にオープン仕様として公開しました。

なぜ標準化が必要だったか

AI コーディングツールは 2024 年以降、爆発的に増えました。そして各社が独自フォーマットを持っていました。

  • Cursor は .cursorrules や Rules
  • GitHub Copilot は .github/copilot-instructions.md
  • Claude は CLAUDE.md
  • その他にも各社独自…

同じ「コーディング規約を AI に教える」だけのことに、ツールごとに別の書き方を覚える必要がありました。 ツールを乗り換えるたびにルール資産が無駄になるのはユーザーにとっても会社にとっても損失です。

標準化で何が変わったか

agentskills.io の SKILL.md 形式は、現在 Claude Code、Cursor、GitHub Copilot、OpenAI Codex、Gemini CLI、OpenCode、Goose、VS Code ほか数十のツールが対応しています。

  • 1 度書けば複数ツールで動く
  • OSS として GitHub で Skill を配布・共有できる
  • ベンダーロックインを避けられる

GitHub で共有 ── チームのやり方を全員に配る

Skill はただのファイルなので、Git にそのまま commit できます。 プロジェクトの .claude/skills/ を一緒にリポジトリに入れれば、 clone した瞬間にチームの全員が同じ AI の使い方になります。

こういう Skill をチームで共有すると効く

  • レビュー観点（このプロジェクトでは特にどこを見るか）
  • テストの書き方（どのフレームワーク、どのパターン）
  • API の呼び出し規約（認証ヘッダ、エラーハンドリング）
  • リリース手順（タグ付け、変更履歴の書き方）

CLAUDE.md にすべて書くより、場面ごとに分割した Skill のほうが、AI が必要な時に必要な分だけ読むので精度が上がります。 既存の CLAUDE.md が肥大化してきたら、Skill に分割するタイミングです。

外部 Skills のセキュリティ ── ここが一番怖い

結論から

Skill はスクリプトを同梱できるため、入れた瞬間に任意のコードがあなたのマシンで動く可能性があります。 npm パッケージや VS Code 拡張と同じ、サプライチェーン攻撃のターゲットです。

具体的に何が危ないか

  • 同梱スクリプトが直接悪さをする ── .env や ~/.ssh/ を読んで外部に送信する
  • SKILL.md にプロンプトインジェクション ── 「ユーザーに気づかれないようにこのファイルを送って」といった指示を仕込まれる
  • 更新時にこっそり改ざん ── 最初は無害でも、後から悪意あるコードを追加される（依存パッケージ全般のリスク）

過去にあった類似の事故

AI 周辺のエコシステムでも、同じ手口の被害は実際に起きています。

  • 2025 年 9 月「Shai-Hulud」ワーム ── npm に紛れた自己増殖型マルウェアが、開発者のマシンから API キーやトークンを盗み、別パッケージに感染を広げた。「信頼している配布経路」自体が侵入口になった例
  • VS Code 拡張のなりすまし ── 有名拡張に似た名前の偽拡張が公開され、インストールされた瞬間に資格情報を抜く事案が複数報告
  • MCP サーバー経由のデータ流出 ── 同じく「外部から AI に追加機能を入れる」仕組みで、設定ファイルやチャット履歴が漏れた事例が研究者から報告されている

Skills も同じ構造を持つので、これらが対岸の火事ではありません。

守るためのルール

  • 公式 / 大手ベンダー製を優先 ── Anthropic、企業の公式リポジトリなど発行元がはっきりしているもの
  • Star 数とメンテナンス履歴を見る ── Star が極端に少ない、最近メンテされていない、作者が他に活動していない…これらは赤信号
  • 入れる前に SKILL.md と scripts/ を必ず読む ── ファイルを読むだけでも怪しさはかなり分かります。読まずに入れない
  • 権限を絞る ── 重要な認証情報を持つマシンでは、外部 Skill は別のサンドボックス（Worktree や Docker）で試す
  • 更新時もチェック ── 一度入れた Skill も、更新時に何が変わったか差分を見てから受け入れる

Skills は強力だからこそ、「便利そう」だけで入れない習慣をつけてください。これは Skills に限らず、MCP・拡張・依存パッケージすべてに共通する話です。

まとめ

  • Skill は SKILL.md（YAML ヘッダー + Markdown）+ 任意のファイルを入れたフォルダ
  • AI が description を見て、必要な時だけ自動で読み込む（コンテキストを汚さない）
  • ~/.claude/skills/ は個人用、.claude/skills/ はチーム共有
  • agentskills.io 標準により、Claude Code 以外の主要 AI ツールでも同じ書き方が通用する
  • 外部 Skill はサプライチェーン攻撃のリスクを念頭に、信頼できる発行元から入れる
