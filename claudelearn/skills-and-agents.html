まず全体像 ― 4つの仕組みを料理で理解する

参照: Skills / Sub-agents / Commands / Agent Skills IO

Claude Codeには、AIの振る舞いをカスタマイズする仕組みが4つあります。
名前が似ていて混乱しがちですが、それぞれ全く違う役割です。まず料理に例えて理解しましょう。

スキル = レシピ本

「カレーの作り方」が書いてあり、「カレー作って」と言われたらAIが自動で開く。材料リスト（リファレンス）も別ページにある。何度でも同じ品質で再現できる。

ルール = キッチンの張り紙

「肉を触ったら必ず手を洗え」。肉を触るときだけ適用される。野菜を切ってるときは関係ない。CLAUDE.mdに「条件」をつけたもの。

コマンド = 口頭の指示メモ

「今日はこのメニューで」と自分で手渡すだけ。AIが勝手に使ってくれることはない。スキルの下位互換で、今後消えていく存在。

サブエージェント = 助手

「皮むきはお前に任せた」と別の人に仕事を渡す。メインの料理人の頭（コンテキスト）を使わない。結果だけが返ってくる。

比較表

スキル ルール コマンド サブエージェント
AIが自動で使う？ 使う 使う（条件一致時） 使わない 使う
手動で呼べる？ 呼べる 呼べない（自動のみ） 呼べる（手動のみ） 呼べる
別ファイル参照 できる できない できない できない
コンテキスト 段階的に読み込み 条件一致時のみ 呼び出し時に全読み 別のコンテキスト
置き場所 .claude/skills/ .claude/rules/ .claude/commands/ .claude/agents/

CLAUDE.mdとの関係

CLAUDE.mdは常に全文が読み込まれます。どんなプロンプトでも必ず知っておいてほしいこと（「bunを使え」「日本語で答えろ」など）を書く場所です。上の4つは「常に読み込む必要がないもの」を効率的に管理する仕組みです。

スキル ― AIへの「頼み方」を再利用する

参照: Skills / Agent Skills Specification

AIに毎回1から説明し直していませんか？
スキルは「AIへの頼み方をファイルに書いて、何度でも再利用する仕組み」です。

スキルがないと何が困るのか

例：結婚式場サイトで10会場分のページを作る仕事

スキルなし

毎回「このページ構成で、この画像を使って、このコンポーネントで…」と1から説明。足りない情報があればやり直し。1会場あたり約2時間。

スキルあり

/add-venue と呼び出すだけ。手順も参照ファイルもスキルに書いてあるので、毎回同じ品質。約5分で261ファイルが自動生成。

スキルの正体：ただのマークダウンファイル

スキルは特別なものではありません。YAMLヘッダー（フロントマター）+ マークダウン本文だけです。

# .claude/skills/compress-image/SKILL.md --- name: compress-image # 必須：スキルの名前 description: > # 必須：AIがこれを見て使うか判断する 500KB以上の画像をsharpパッケージで圧縮・最適化する。 画像圧縮、画像最適化、large images、compress image などのキーワードで使用する。 --- ## ワークフロー 1. プロジェクト内の画像ファイルを走査 2. 500KB以上のファイルを特定 3. scripts/compress-image.ts で圧縮を実行 4. 圧縮結果をレポート ## 圧縮スクリプト 画像の圧縮には scripts/compress-image.ts を使用すること。

name と description だけが必須です。descriptionに書いたキーワード（「画像圧縮」など）をプロンプトで使うと、AIが自動でこのスキルを呼び出します。

ディレクトリ構成

スキルは単一ファイルではなくディレクトリです。リファレンスやスクリプトを別ファイルとして持てます。

.claude/skills/ compress-image/ # スキルのディレクトリ SKILL.md # 本体（これが必須） scripts/ # スクリプト（任意） compress-image.ts # 実際の圧縮処理 references/ # 参照ファイル（任意） supported-formats.md # 対応フォーマットの説明など

Progressive Disclosure（段階的開示）― スキル最大の特徴

CLAUDE.mdとスキルの最大の違いがここです。CLAUDE.mdは常に全文が読み込まれますが、スキルは必要なときだけ段階的に読み込まれます。

1
メタデータだけが常にAIに見えている
name と description の数行だけ。10個スキルがあっても数十行程度しかコンテキストを使わない。
↓ 「画像を圧縮して」→ AIが「compress-imageスキルが使えそう」と判断
2
SKILL.mdの本文を読み込む
ワークフロー、注意事項など具体的な手順がここで初めてコンテキストに入る。
↓ 「圧縮スクリプトが必要」→ リファレンスを参照
3
リファレンス・スクリプトを読み込む
コードテンプレート、型定義、実行スクリプトなど補助ファイルを必要に応じて参照。

なぜこれが重要なのか

コンテキストウィンドウが大きくなるほどAIの精度は落ちます。CLAUDE.mdに全部書くと、関係ない作業でもその情報が読み込まれてコンテキストを圧迫します。スキルなら必要なときだけ必要な情報が読み込まれるので、コンテキストを効率的に使えます。

スキルの作り方

自分で1から書く必要はありません。AIに作らせます。

1
Skill Creatorを入れる（初回のみ）
npx skillsz
対話式UIで、使っているエージェント（Claude Code等）を選択 → グローバル/プロジェクト選択 → 完了
2
Claude Code内でスキルを作成
/skill-creator
「500KB以上の画像を圧縮するスキルを作って」と伝えるだけ。AIがSKILL.md、リファレンス、スクリプトまで全部作ってくれる。

スキルの入手先：skills.sh

自分で作らなくても、skills.sh（Vercel運営のスキルマーケットプレイス）から高品質なスキルをインストールできます。npmのスキル版です。

$ npx skillsz # 対話式でインストール $ npx skillsz list # インストール済み一覧 $ npx skillsz update # 最新版に更新

外部スキルのセキュリティに注意

公開スキルの36.8%にセキュリティ上の問題があるという調査データがあります。スキルの中に rm -rf / のような危険なコマンドが書かれていたら、AIはそれを実行しようとします。
skills.shにはaudit（監査）機能があるので確認しましょう。インストール数が極端に少ないものは特に注意してください。

ルール ― 「このファイルを触るときだけ」適用される指示

参照: Memory

ルールはCLAUDE.mdに「条件」をつけたものです。
「.tsxファイルを触るときだけこの指示に従え」のように、globパターンで適用範囲を絞れます。

なぜルールが必要なのか ― 具体例

Tailwind CSSを使ったプロジェクトで「font-medium、font-semiboldは使うな（日本語フォントはnormalかboldだけ）」という指示を出したいとします。

CLAUDE.mdに書いた場合

README.mdを編集してるときも、package.jsonを触ってるときも、この指示が読み込まれる。Tailwindと無関係なのにコンテキストを消費。

ルールにした場合

.tsx .jsx .css ファイルを触るときだけ読み込まれる。他のファイル作業ではコンテキストを消費しない。

ルールファイルの例

# .claude/rules/tailwind.md --- globs: ["*.tsx", "*.jsx", "*.css"] # この拡張子のファイルを触るときだけ適用 alwaysApply: false # falseなら条件一致時のみ --- ## Tailwind CSS ルール - font-medium, font-semibold は使わないこと （日本語フォントはnormalかboldのみ） - spacing は 4の倍数を基本とすること - カスタムカラーは tailwind.config.ts に定義してから使うこと

CLAUDE.md / ルール / スキルの使い分け

  • CLAUDE.md → どんな作業でも常に知っておいてほしいこと（「bunを使え」「日本語で答えろ」）
  • ルール → 特定のファイルを触るときだけ必要なこと（「.tsxではfont-medium使うな」）
  • スキル → 繰り返す作業の手順書（「会場を追加するときはこの手順で」）

コマンド ― 手動で呼ぶだけのシンプルな指示（スキルに統合済み）

参照: Commands

コマンドは手動でのみ呼び出せる単一ファイルの指示です。
Claude Codeではスキルとカスタムスラッシュコマンドが統合されたため、新しく作るならスキルを使いましょう。

コマンドとスキルの違い

コマンド スキル
AIが自動で呼ぶ できない できる
別ファイル参照 できない（単一ファイル） できる（ディレクトリ構成）
段階的読み込み なし あり
今後の方向性 スキルに吸収されていく これからの主役

既存のコマンドがある場合は、順次スキルに移行するのがおすすめです。

サブエージェント ― 別のAIに仕事を任せる

参照: Sub-agents

サブエージェントは特定のタスク専用の別のAIです。
メインのAIとは完全に別のコンテキストで動くため、メインの記憶を一切消費しません。

なぜサブエージェントが必要なのか

メインのAIに全部やらせると、コンテキストがどんどん消費されて品質が落ちます。サブエージェントは「仕事を分担する」仕組みです。

メインエージェント
プランを立てて、タスクを振り分ける
↓ 委任 ↓ 委任 ↓ 委任
code-reviewer
コードレビュー専用
security-check
セキュリティ監査専用
refactorer
リファクタリング専用

スキルとサブエージェントの違い

どちらもAIにタスクをさせる仕組みですが、「誰が実行するか」が決定的に違います。

スキル サブエージェント
誰が実行？ メインのAIが手順に従って自分で実行 別のAIが独立して実行
コンテキスト メインのコンテキストを使う 独立したコンテキストで動く
向いてる仕事 手順が決まった繰り返し作業 独立した分析・レビュー・並列作業

Claude Codeの組み込みサブエージェント

  • Explore ― コードベース検索専用（Haiku、読み取り専用、高速）
  • Plan ― 計画・調査専用（読み取り専用）
  • general-purpose ― 汎用的な複雑処理（全ツール利用可）

相互呼び出しができる

スキルからサブエージェントを呼び出す、サブエージェントがスキルを使う、といった相互連携が可能です。例えば「会場追加スキル」の中で「セキュリティチェック用サブエージェント」を呼び出して、生成したコードを自動レビューさせることもできます。

プラグイン ― スキル・ルール・サブエージェントのまとめ売り

参照: Plugins

スキル・ルール・コマンド・サブエージェントを1つずつ揃えるのは大変です。
プラグインはこれらをまとめてインストールできるパッケージです。

結局どうすればいい？

  • 汎用的な作業（ブレスト、プラン、レビューなど）→ プラグインや skills.sh から入れる。先人が作った高品質なものの方が自作より良い
  • ドメイン固有の作業（会場追加、業務特化の処理など）→ /skill-creator で自分で作る。野良に転がっているわけがない
  • 特定ファイルの制約（Tailwindルール、テスト規約など）→ .claude/rules/ にルールとして配置
