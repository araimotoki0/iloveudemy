ベストプラクティス ー 最も重要な原則 Docs

Claude Codeを効果的に使うために、最も重要な原則があります。 それはコンテキストウィンドウは有限であり、埋まるほどパフォーマンスが低下するということです。 この制約を前提に、以下の5つの原則を実践してください。

核心的な制約

コンテキストウィンドウはすぐに埋まります。埋まるにつれてLLMの性能は劣化し、 指示の見落としや品質の低下が発生します。以下の5原則はすべて、この制約を前提にしたものです。

1

検証手段を与える

Claudeに自分の作業を検証する方法を必ず提供してください。 テスト、スクリーンショット、期待される出力などがあれば、 Claudeは自律的にイテレーションして正解にたどり着けます。

改善前（曖昧） 改善後（検証可能）
implement a function that validates email write validateEmail. test cases: user@example.com=true, invalid=false, @bad.com=false. run tests after
make the dashboard look better [スクリーンショットを貼り付け] implement this design. take a screenshot after and compare
the build is failing the build fails with this error: [エラー貼付]. Fix it, verify the build passes. Address root cause, not symptoms.

検証がもたらす効果

検証手段があると、Claudeはエージェントループの中で自ら「テスト実行 → 失敗確認 → 修正 → 再テスト」を繰り返せます。 検証がなければ、Claudeは「たぶん正しい」コードを返すだけで終わります。

|
2

先に探索、次に計画、そしてコーディング

複雑なタスクは、いきなりコーディングに入らず段階的に進めましょう。

探索
Plan Mode
Phase 2
計画
Plan Mode
Phase 3
実装
Normal Mode
Phase 4
コミット
Normal Mode

計画をスキップできる場合：スコープが明確で小さい修正（タイポ修正、既知のバグフィックスなど）

|
3

具体的なコンテキストを提供する

曖昧なプロンプトは探索範囲を広げ、コンテキストを無駄に消費します。 具体的に指示すればするほど、効率的かつ正確になります。

改善前 改善後
add tests for foo.py write test for foo.py covering edge case where user is logged out. avoid mocks.
why does ExecutionFactory have weird api? look through ExecutionFactory's git history and summarize how its api came to be
add a calendar widget look at how existing widgets are implemented. HotDogWidget.php is a good example. follow that pattern to add CalendarWidget.
fix the login bug users report login fails after session timeout. check auth flow in src/auth/ and session management in src/session/
|
4

リッチなコンテンツを提供する

Claude Codeはテキストだけでなく、様々な形式のコンテンツを理解できます。 積極的に活用しましょう。

@ファイル参照

スクリーンショットやデザインモックをそのままペースト

APIドキュメントやIssueのURLを渡してClaudeにフェッチさせる

パイプでデータを渡す

cat logs.txt | claude -p "分析して"

5

セッションを積極的に管理する

コンテキストウィンドウは有限です。積極的にセッションを管理して、常にクリーンな状態を保ちましょう。

Esc

現在の応答を停止

Esc + Esc

最後のターンを巻き戻し（Undo）

"undo that"

変更を元に戻すようClaudeに指示

/clear

タスク間でコンテキストをリセット

2回以上修正したら /clear

同じ問題で2回以上Claudeを修正している場合、コンテキストが汚れています。 /clear してプロンプトを書き直す方が、修正を重ねるより速くて正確です。

サブエージェント Docs

サブエージェントは、メインのClaudeセッションとは独立した別のエージェントです。 独自のコンテキストウィンドウ、ツール、システムプロンプトを持ちます。

なぜサブエージェントを使うのか？

コンテキスト隔離

大量のファイルを探索しても、メインのコンテキストを汚さない。サブエージェントが要約だけを返す。

並列処理

複数のサブエージェントが同時に異なるタスクを実行できる。

専門化

セキュリティレビュー、テスト生成など、特定の役割に特化した設定ができる。

カスタムサブエージェントの作成

.claude/agents/ ディレクトリにMarkdownファイルを作成します。 フロントマターでエージェントの設定を定義できます。

# .claude/agents/security-reviewer.md

---
name: Security Reviewer
description: セキュリティの観点からコードをレビューする
tools: [Read, Grep, Glob]
model: claude-sonnet-4-20250514
isolation: worktree
---

あなたはセキュリティレビューの専門家です。
以下の観点でコードをレビューしてください：
- SQLインジェクション
- XSS脆弱性
- 認証・認可の問題
- 機密情報の露出

フロントマターの主要フィールド

フィールド 説明
name エージェントの表示名
description Claudeが自動委譲する際の判断材料になる説明文
tools 使用可能なツールのリスト
model 使用するモデル（sonnet / opus など）
allowed-tools 許可するツールの細かい指定
isolation worktree で独立したGit worktreeで作業
skills エージェントが持つスキル定義
auto-memory 自動メモリ機能の有効化

ビルトインエージェントタイプ

Explore

読み取り専用。コードベースの調査・分析に使用。ファイルの変更はできない。

Plan

アーキテクチャの設計・計画に特化。実装方針を立てる。

汎用エージェント

すべてのツールにアクセスできる一般的なサブエージェント。

サブエージェントの使い方

Claudeは description フィールドに基づいて自動的にサブエージェントに委譲します。明示的に指示することもできます。

use a subagent to review this code for security issues
/agents
利用可能なエージェント一覧を表示し、管理できます

Worktree隔離

isolation: worktree とは

isolation: worktree を設定すると、サブエージェントはGit worktreeで独立したディレクトリで作業します。 メインの作業ディレクトリとファイルの競合が発生しないため、安全に並列作業ができます。

コンテキスト分離のメリット

サブエージェントなし
50ファイル読む → メインコンテキスト肥大化 → パフォーマンス低下
サブエージェントあり
サブが50ファイル探索 → 要約3行を返す → メインコンテキストはクリーン

並列実行とスケーリング Docs Docs

Claude Codeは単一セッションだけでなく、複数のセッションやエージェントを並列に動かすことで 大規模なタスクに対応できます。

Git Worktrees

ワークツリーを使うと、同じリポジトリの異なるブランチを同時に操作できます。

claude --worktree feature-auth
新しいworktreeとブランチ feature-auth を作成し、そこでClaudeを起動
claude -w
自動的にworktreeを作成（短縮形）

ワークツリーは .claude/worktrees/<name>/ に作成され、 新しいブランチが自動的に切られます。変更がなければ自動クリーンアップされ、 変更がある場合はプロンプトが表示されます。

複数セッションの活用

デスクトップアプリ

複数のローカルセッションを視覚的に管理できる。

Web版（claude.ai/code）

クラウドVM上で実行。長時間タスクやリモート作業に最適。

エージェントチーム

複数のエージェントが自動的に協調して作業する。

Writer/Reviewerパターン

セッションA（Writer）
コードを書く・機能を実装する
コミット&プッシュ
セッションB（Reviewer）
クリーンなコンテキストでレビュー

Fan-outパターン

タスクリストを生成し、ファイルごとに個別のClaudeセッションをループで実行するパターンです。

# タスクリストを生成
claude -p "list all files needing migration" > tasks.txt

# 各ファイルに対して並列実行
while IFS= read -r file; do
  claude -p "migrate $file to new API" \
    --allowedTools "Read,Edit,Write" &
done < tasks.txt
wait

非インタラクティブモード（Print Mode）

claude -p でスクリプトやCI/CDパイプラインから使えます。

claude -p "この関数を説明して" --output-format json
cat error.log | claude -p "このエラーの原因を分析して" > analysis.txt
出力形式 用途
--output-format text プレーンテキスト出力（デフォルト）
--output-format json JSON構造化出力。スクリプトでパースしやすい
--output-format stream-json JSONストリーミング。リアルタイム処理向け

コンテキストウィンドウ管理 Docs Docs

コンテキストウィンドウは、会話全体、読み取ったファイル、コマンド出力のすべてを保持する空間です。 この空間が埋まるにつれてLLMの性能が低下するため、積極的な管理が不可欠です。

管理戦略

1
/clear ー タスク間のリセット

無関係なタスクに移る前に /clear でコンテキストをリセット。 最もシンプルで効果的な管理方法です。

|
2
/compact ー 会話の要約圧縮

現在の会話を要約して圧縮します。フォーカス指示も可能：

/compact Focus on API changes
|
3
Esc+Esc → ここから要約（部分圧縮）

Escを2回押して巻き戻し、「ここから要約して」と指示。 会話の一部だけを圧縮できます。

|
4
/btw ー サイドクエスチョン

一時的な質問に使用。エフェメラル（一時的）で、メインのコンテキストを消費しません。

/btw TypeScriptでenumを使うべきか、union typeを使うべきか？
|
5
サブエージェントで調査を委譲

大量のファイルを読む調査タスクはサブエージェントに委譲。 メインのコンテキストには要約だけが返ります。

CLAUDE.mdでの圧縮指示

When compacting, always preserve:
- List of modified files
- Current task status
- Key architectural decisions made

自動圧縮（Auto-compaction）

自動的にトリガーされる圧縮

コンテキストウィンドウが限界に近づくと、Claude Codeは自動的に会話を要約・圧縮します。 重要なコード変更や決定事項は保持されますが、意図的に管理する方が品質は高くなります。

コスト管理 Docs

Claude Codeはエージェントループの特性上、通常のチャットよりも多くのトークンを消費します。 コストを意識した使い方が重要です。

なぜClaude Codeはトークンを多く使うのか

エージェントループの構造的コスト

エージェントループでは、各イテレーションごとに会話履歴全体がAPIに送信されます。 ループが10回回れば、初期メッセージも10回送信されることになります。 これがClaude Codeのトークン消費が多い根本的な理由です。

コスト確認コマンド

/cost
現在のセッションのトークン使用量とコスト
/usage
プランの使用制限と残りの利用可能量

コスト削減戦略

/clear を頻繁に使う

コンテキストをリセットすることで、次のイテレーションで送信されるトークン数を削減。

トラブルシューティング Docs

Claude Codeの使用中によく発生する問題パターンと、その対処法を解説します。

よくある失敗パターン

1. キッチンシンクセッション

症状：無関係なタスクを同じセッションで次々と実行し、コンテキストが膨張。

対策：タスクが変わるたびに /clear を実行。 「認証のバグ修正」と「UIの改善」は別セッションで行う。

2. 繰り返し修正

症状：同じ問題に対して何度もClaudeを修正している。

対策：/clear してプロンプトを最初から書き直す。 修正の積み重ねよりも、良いプロンプトの方が効果的。

3. CLAUDE.mdの肥大化

症状：CLAUDE.mdが長くなりすぎて、Claudeがルールを無視し始める。

対策：定期的にCLAUDE.mdを見直し、不要なルールを削除。 本当に重要なルールだけを残す。サブディレクトリの CLAUDE.md を活用してスコープを絞る。

4. 信頼して検証しないギャップ

症状：Claudeの出力をそのまま受け入れて、後から問題が見つかる。

対策：プロンプトに必ず検証ステップを含める。テスト実行、ビルド確認、スクリーンショット比較など。

5. 無限探索

症状：Claudeが際限なくファイルを読み続け、コンテキストが埋まる。

対策：探索範囲を具体的に指定するか、サブエージェントに委譲する。 「src/auth/ の中だけ調べて」のようにスコープを絞る。

まとめ

ここまでの講座で、Claude Codeの内部メカニズムから実践的な活用法まで、 包括的に学んできました。最後に、全体を振り返りましょう。

学んだ主要トピック

エージェントループ

Claude Codeの核心 ー ユーザー入力 → LLM推論 → ツール実行 → 結果返却の繰り返し

ツールシステム

JSONスキーマで定義されたツールをLLMが選択・呼び出す仕組み

システムプロンプト

Claudeの行動を制御する最上位の指示。CLAUDE.mdによるカスタマイズ

コンテキストウィンドウ

有限のメモリ空間。管理しなければパフォーマンスが劣化する

拡張ポイント

MCP、Hooks、サブエージェント、プラグインによる機能拡張

実践活用

ベストプラクティス、コスト管理、並列実行、トラブルシューティング

最終チェックリスト

以下の項目を確認して、講座の内容を振り返ってください。

基礎理解
- エージェントループの4ステップ（入力 → 推論 → ツール実行 → 結果返却）を説明できる
- ツールがJSONスキーマで定義される仕組みを理解している
- システムプロンプトの構成（不変部分 + CLAUDE.md + 動的情報）を把握している
- コンテキストウィンドウの制約と、それが性能に与える影響を理解している

実践スキル
- CLAUDE.mdを適切に設定できる
- 検証手段を含むプロンプトを書ける
- /clear、/compact、/btw を使い分けられる
- サブエージェントの概念と作成方法を知っている
- コスト意識を持って Claude Code を使える

応用力
- 並列実行パターン（worktree、fan-out）を活用できる
- トラブルシューティングの方法を知っている
- MCPサーバーを設定してツールを拡張できる
- Hooksを使ってワークフローを自動化できる
- 公式ドキュメントから必要な情報を見つけられる

講座を完了しました

Claude Codeの仕組みを理解することで、ただ使うだけでなく、 なぜそう動くのかを理解した上で活用できるようになります。 この知識は、AIエージェントツール全般に応用が効きます。

まずは自分のプロジェクトで小さなタスクから試してみてください。 CLAUDE.md の設定、/clear によるコンテキスト管理、 検証付きプロンプトの作成 ー これらを実践するだけで、Claude Codeの効果は大幅に向上します。
