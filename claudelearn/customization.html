CLAUDE.md ー プロジェクトの記憶 Docs

CLAUDE.md は、Claude Code がセッション開始時に自動的に読み込む Markdown ファイルです。プロジェクト固有のルール、コーディング規約、アーキテクチャの決定事項などを記載しておくことで、毎回同じ指示を繰り返す必要がなくなります。いわば「プロジェクトの記憶」として機能します。

2つのメモリシステム

Claude Code には2種類のメモリシステムがあります。

項目 CLAUDE.md（手動メモリ） Auto Memory（自動メモリ）
誰が書くか あなた（開発者） Claude 自身
内容 プロジェクトルール、規約、コマンド ユーザーの好み、過去の文脈
スコープ プロジェクト / ユーザー / 管理者 プロジェクト単位
読み込み先 システムプロンプトに注入 システムプロンプトに注入
用途 チーム全体で共有するルール 個人の好み・学習内容

CLAUDE.md の配置場所

CLAUDE.md はスコープに応じて複数の場所に配置できます。Claude Code は作業ディレクトリからルートに向かってディレクトリツリーを遡りながら、すべての CLAUDE.md を読み込みます。

スコープ 配置場所 目的 共有
管理者ポリシー（macOS） /Library/Application Support/ClaudeCode/CLAUDE.md 組織全体の強制ルール 組織内全員
管理者ポリシー（Linux） /etc/claude-code/CLAUDE.md 組織全体の強制ルール 組織内全員
管理者ポリシー（Windows） C:\Program Files\ClaudeCode\CLAUDE.md 組織全体の強制ルール 組織内全員
ユーザー ~/.claude/CLAUDE.md 個人の全プロジェクト共通ルール 自分のみ
プロジェクト（ルート） ./CLAUDE.md プロジェクト固有のルール チーム全員（Git管理）
プロジェクト（隠しディレクトリ） ./.claude/CLAUDE.md プロジェクト固有のルール チーム全員（Git管理）

効果的な CLAUDE.md の書き方

ベストプラクティス

  • サイズ：200行以下を目標にする。長すぎるとコンテキストウィンドウを圧迫する
  • 構造：Markdown の見出し（#、##）と箇条書きで整理する
  • 具体性：検証可能な具体的指示を書く（「2スペースインデント」のように）
  • 一貫性：矛盾する指示を避ける

インポート構文

@path/to/file 構文で、外部ファイルの内容を CLAUDE.md に取り込めます。

# CLAUDE.md の中で他のファイルをインポート @README.md # プロジェクトルートの README @package.json # 依存関係情報 @docs/architecture.md # アーキテクチャドキュメント @~/.claude/my-project-instructions.md # ユーザーレベルの指示

インポートの制限

  • 相対パスと絶対パスの両方が使用可能
  • 最大深度は 5ホップ（インポートのインポートを含む）
  • 循環参照は自動的に検出・防止される

/init コマンド

/init コマンドを実行すると、Claude がコードベースを分析し、最適な CLAUDE.md を自動生成します。

/init
Claude がプロジェクト構造を分析し、言語、フレームワーク、
テストコマンド、コーディング規約などを検出して
CLAUDE.md のひな型を生成します。

CLAUDE.md の読み込み順序

Claude Code は作業ディレクトリからルートディレクトリに向かってディレクトリツリーを遡り、見つかったすべての CLAUDE.md を読み込みます。上位のファイルほど優先度が低くなります。

1 管理者ポリシー CLAUDE.md（最高優先度）
↓
2 ユーザー ~/.claude/CLAUDE.md
↓
3 プロジェクトルート ./CLAUDE.md または ./.claude/CLAUDE.md
↓
4 サブディレクトリ（作業ディレクトリまで遡る）

書くべき内容 vs 書くべきでない内容

書くべき内容 書くべきでない内容
Claude が推測できない bash コマンド コードから読み取れる情報
コードスタイルのルール 標準的な慣習（自明なもの）
テストの実行方法 詳細な API ドキュメント
リポジトリの作法（PR ルールなど） 頻繁に変わる情報
アーキテクチャ上の重要な決定事項 ファイルごとの詳細な説明
開発環境の特殊な設定 自明なプラクティス
よくあるハマりどころ・注意点 一般的なプログラミング知識

CLAUDE.md の良い例

# プロジェクト概要 TypeScript + React のモノレポ。pnpm を使用。 # 開発コマンド - ビルド: pnpm build - テスト: pnpm test（単体テスト1つ: pnpm test -- path/to/test） - リント: pnpm lint # コーディング規約 - インデント: 2スペース - import は絶対パスを使う（@/components/Button） - コンポーネントは名前付きエクスポート（default export 禁止） - テストファイルは *.test.ts の命名規則 # アーキテクチャ - src/api/ - バックエンド API（Express） - src/web/ - フロントエンド（React） - src/shared/ - 共通型定義とユーティリティ

.claude/rules/ ー ルールファイル Docs

ルールファイルは、.claude/rules/ ディレクトリに配置するモジュール式の指示ファイルです。CLAUDE.md を一枚の大きなファイルにする代わりに、関心ごとに分割して管理できます。特定のファイルパスにのみ適用されるルールも定義可能です。

ディレクトリ構造の例

.claude/rules/ ├── code-style.md # 全体のコードスタイル ├── testing.md # テストに関するルール ├── api-conventions.md # API 設計の規約 ├── react-components.md # React コンポーネントのルール └── security.md # セキュリティ関連のガイドライン

パス固有のルール（YAML フロントマター）

ルールファイルに YAML フロントマターを追加すると、特定のパスに一致するファイルが会話に含まれる場合にのみそのルールが適用されます。

--- paths: ["src/api/**/*.ts"] --- # API ルール - すべてのエンドポイントに入力バリデーションを実装すること - エラーレスポンスは RFC 7807 形式に従うこと - 認証ミドルウェアを必ず適用すること

Glob パターンの例

パターン マッチ対象
**/*.ts すべての TypeScript ファイル
src/**/* src ディレクトリ以下のすべてのファイル
*.md ルート直下の Markdown ファイル
src/components/*.tsx components 直下の TSX ファイル
tests/**/*.test.ts tests 以下のテストファイル

ユーザーレベルのルール

~/.claude/rules/ にルールファイルを配置すると、すべてのプロジェクトに適用される個人ルールになります。

~/.claude/rules/ ├── personal-style.md # 個人のコーディングスタイル └── always-japanese.md # 「常に日本語で応答」など

シンボリックリンクによる共有

シンボリックリンクを使えば、複数プロジェクト間でルールファイルを共有できます。

ln -s ~/shared-rules/security.md .claude/rules/security.md

ルール vs CLAUDE.md：いつどちらを使うか

  • CLAUDE.md：プロジェクト全体に適用される基本ルール、ビルドコマンド、アーキテクチャ概要
  • ルールファイル：特定の技術領域やファイルパスに限定されるルール、頻繁に更新されるルール、チーム間で共有・再利用したいルール

Auto Memory ー 自動メモリ Docs

Auto Memory は、Claude が自分自身のためにメモを保存する機能です。セッション中に学んだユーザーの好みやプロジェクトの特徴を記録し、次回以降のセッションで活用します。

有効化・無効化

  • /memory コマンドで切り替え
  • 設定ファイルで autoMemoryEnabled を true / false に設定

保存場所

~/.claude/projects/<project-hash>/memory/ ├── MEMORY.md # インデックスファイル（最初の200行が常に読み込まれる） ├── coding-preferences.md # トピック別ファイル ├── architecture-notes.md # トピック別ファイル └── debugging-tips.md # トピック別ファイル

仕組み

1 MEMORY.md（インデックス）の最初の200行がセッション開始時に読み込まれる
↓
2 関連するトピックファイルはオンデマンドで読み込まれる
↓
3 セッション中に新しい学びがあれば、Claude が自動的にメモリを更新

/memory コマンド

/memory コマンドで、保存されたメモリの閲覧・編集ができます。

/memory
メモリの一覧表示、個別のメモリの確認・編集・削除が可能

手動でのメモリ操作

  • 「これを覚えておいて」（Remember this）と伝えると、Claude がメモリに保存
  • 「これは忘れて」（Forget this）と伝えると、Claude が該当するメモリを削除
  • メモリファイルはプレーンテキストなので、直接エディタで編集することも可能

注意点

Auto Memory は個人単位で保存されるため、チームメンバーとは共有されません。チーム共有のルールは CLAUDE.md やルールファイルを使用してください。

Skills ー カスタムコマンド Docs

Skills は、SKILL.md ファイルを使って Claude Code の機能を拡張するカスタムコマンドです。スラッシュコマンド（/my-skill）として呼び出すことも、Claude が自動的に適切なタイミングで使用することもできます。

Skills の配置場所

スコープ 配置場所 説明
Enterprise 管理者設定で配信 組織全体で共有されるスキル
Personal ~/.claude/skills/ 個人の全プロジェクトで使えるスキル
Project .claude/skills/ プロジェクト固有のスキル（Git管理可）
Plugin npm パッケージ経由 外部パッケージとして配布されるスキル

Skill の作成

ディレクトリを作成し、その中に SKILL.md ファイルを置きます。

.claude/skills/ └── deploy/ ├── SKILL.md # スキル定義（必須） └── deploy-checklist.md # サポートファイル（任意）

SKILL.md の構造

--- name: deploy description: 本番環境へのデプロイを実行する argument-hint: [staging|production] --- # デプロイ手順 1. まずテストを実行: `pnpm test` 2. ビルドを確認: `pnpm build` 3. 引数に応じて環境を選択: - staging: `pnpm deploy:staging` - production: `pnpm deploy:prod` 4. デプロイ結果を確認して報告

フロントマターの全フィールド

フィールド 型 説明
name string スキル名（スラッシュコマンドの名前になる）
description string スキルの説明。Claude がいつ使うか判断する材料になる
argument-hint string 引数のヒント（UI に表示される）
disable-model-invocation boolean true にすると Claude が自動呼び出しできなくなる（ユーザー専用）
user-invocable boolean false にするとユーザーからは呼び出せない（Claude 専用）
allowed-tools string[] スキルが使用できるツールのリスト
model string スキル実行に使用するモデル
context string fork でサブエージェントとして実行
agent string 実行するエージェントの指定
hooks object スキル固有のフック定義

文字列置換（変数）

変数 説明
$ARGUMENTS スキルに渡されたすべての引数（文字列全体）
$ARGUMENTS[N] N番目の引数（0ベース）
$N N番目の引数のショートカット（$ARGUMENTS[N] と同等）
${CLAUDE_SESSION_ID} 現在のセッション ID
${CLAUDE_SKILL_DIR} SKILL.md があるディレクトリのパス

ビルトインスキル

/simplify

コードを簡潔にリファクタリング

/batch

複数ファイルに同じ操作を適用

/debug

エラーの原因を調査・修正

/loop

テストが通るまで繰り返し修正

/claude-api

Claude API を直接呼び出す

呼び出し制御

モード 設定 説明
両方（デフォルト） 設定なし ユーザーも Claude も呼び出せる
ユーザー専用 disable-model-invocation: true ユーザーのみがスラッシュコマンドで呼び出せる
Claude 専用 user-invocable: false Claude が適切なタイミングで自動的に使用

高度なパターン

動的コンテキスト（! 構文）

バッククォートの前に ! を付けると、コマンドの実行結果がスキルのコンテキストに含まれます。

# 現在のブランチ情報を動的に取得 現在のブランチ: !`git branch --show-current` 最新のコミット: !`git log --oneline -5`

サブエージェントとして実行

context: fork を設定すると、スキルは独立したサブエージェントとして実行されます。メインの会話コンテキストを汚さずに独立した作業を行えます。

--- name: analyze context: fork --- # サブエージェントとして独立実行される

サポートファイル

スキルのディレクトリにテンプレートやチェックリストなどのファイルを配置し、${CLAUDE_SKILL_DIR} で参照できます。

Hooks ー 自動化フック Docs

Hooks は、Claude Code のライフサイクルの特定のポイントで自動的に実行されるシェルコマンド、HTTP リクエスト、プロンプト、またはエージェントです。コードのフォーマット、危険な操作のブロック、通知の送信など、ワークフローの自動化に使用します。

フックイベント一覧

カテゴリ イベント名 タイミング
セッション SessionStart セッション開始時
SessionEnd セッション終了時
InstructionsLoaded CLAUDE.md/ルール読み込み完了時
ユーザー UserPromptSubmit ユーザーがプロンプトを送信した時
ツール PreToolUse ツール実行前
PostToolUse ツール実行後（成功時）
PostToolUseFailure ツール実行後（失敗時）
エージェント SubagentStart サブエージェント起動時
SubagentStop サブエージェント停止時
Stop Claude がレスポンスを完了した時
TeammateIdle チームメイトがアイドル状態になった時
TaskCompleted タスクが完了した時
その他 Notification 通知が発生した時
ConfigChange 設定が変更された時
WorktreeCreate ワークツリー作成時
WorktreeRemove ワークツリー削除時
PreCompact コンパクト処理前
PostCompact コンパクト処理後
Elicitation ユーザーへの質問時
権限 PermissionRequest 権限の承認リクエスト時

ハンドラーの種類

command

シェルコマンドを実行

http

HTTP リクエストを送信

prompt

プロンプトを注入

agent

エージェントを起動

設定場所

Hooks は settings.json ファイルに定義します。ユーザー、プロジェクト、ローカル、管理者の各スコープで設定可能です。

終了コード

  • 0 - 成功。処理を続行
  • 2 - ブロッキングエラー。操作をキャンセル（PreToolUse の場合、ツール実行を阻止）
  • その他 - 非ブロッキングエラー。警告を表示して続行

マッチャーパターン

matcher フィールドで、特定のツールに限定してフックを発火させます。

// Edit ツールのみにマッチ "matcher": "Edit" // Bash ツールのみにマッチ "matcher": "Bash" // 複数ツールにマッチ（正規表現） "matcher": "Edit|Write"

実践例 1：ファイル編集後に自動フォーマット

{ "hooks": { "PostToolUse": [ { "matcher": "Edit", "hooks": [ { "type": "command", "command": "npx prettier --write $CLAUDE_FILE_PATH" } ] } ] } }

実践例 2：危険なコマンドをブロック

{ "hooks": { "PreToolUse": [ { "matcher": "Bash", "hooks": [ { "type": "command", "command": "if echo \"$CLAUDE_TOOL_INPUT\" | grep -q 'rm -rf /'; then echo 'BLOCKED: 危険なコマンド' >&2; exit 2; fi" } ] } ] } }

終了コード 2 を返すことで、ツール実行がブロックされます。

実践例 3：デスクトップ通知

{ "hooks": { "Notification": [ { "hooks": [ { "type": "command", "command": "osascript -e 'display notification \"$CLAUDE_NOTIFICATION\" with title \"Claude Code\"'" } ] } ] } }

MCP ー 外部ツール連携 Docs

MCP（Model Context Protocol）は、AI ツール連携のオープン標準プロトコルです。課題管理、データベース、監視ツール、デザインツール、ワークフロー自動化など、外部サービスと Claude Code を接続します。

MCP サーバーの追加方法

HTTP トランスポート

claude mcp add --transport http <name> <url>

リモートサーバーへの HTTP 接続。最もシンプルな方法。

SSE トランスポート（非推奨）

claude mcp add --transport sse <name> <url>

Server-Sent Events。HTTP トランスポートへの移行が推奨。

stdio トランスポート

claude mcp add --transport stdio <name> -- <command>

ローカルプロセスとして起動し、標準入出力で通信。

MCP サーバーの管理

claude mcp list
設定済みの MCP サーバーの一覧を表示
claude mcp get <name>
特定の MCP サーバーの詳細情報を表示
claude mcp remove <name>
MCP サーバーを削除
/mcp
インタラクティブな MCP 管理画面を開く

スコープ

スコープ 保存先 用途
local（デフォルト） ~/.claude/ 配下のローカル設定 個人の開発環境でのみ使用するサーバー
project .mcp.json（リポジトリルート） チームで共有するサーバー（Git管理）
user ~/.claude/ 配下のユーザー設定 全プロジェクトで使用するサーバー
claude mcp add --scope project --transport stdio github -- npx @modelcontextprotocol/server-github
プロジェクトスコープで GitHub MCP サーバーを追加

OAuth 認証

OAuth 対応の MCP サーバーは、/mcp コマンドから認証フローを開始できます。ブラウザが開き、認証後にトークンが自動的に保存されます。

人気の MCP サーバー

GitHub

Issue、PR、リポジトリ操作

Sentry

エラー監視・デバッグ

Notion

ドキュメント・ナレッジベース

PostgreSQL

データベースクエリ・スキーマ

MCP リソース

MCP サーバーが公開するリソースは、@ メンション構文で Claude の会話に取り込めます。データベースのスキーマ、API ドキュメント、設定ファイルなどを直接参照できます。

MCP プロンプト

MCP サーバーが提供するプロンプトは、/mcp__<server>__<prompt> 形式のコマンドとして利用可能です。

Claude Code を MCP サーバーとして公開

claude mcp serve
Claude Code 自体を MCP サーバーとして起動。
他の AI ツールから Claude Code の機能を利用可能に。

設定ファイル体系 Docs

Claude Code の設定は階層的なスコープで管理されます。各スコープは異なる目的と権限レベルを持ち、設定の優先度が明確に定義されています。

設定スコープ

スコープ 場所 管理者 共有
Managed（管理者） サーバー / plist / レジストリ / managed-settings.json 組織の管理者 組織全体
User（ユーザー） ~/.claude/settings.json 個人 個人のみ
Project（プロジェクト） .claude/settings.json チーム Git管理（チーム共有）
Local（ローカル） .claude/settings.local.json 個人 個人のみ（.gitignore）

設定ファイルの詳細

~/.claude/settings.json ユーザー設定

個人の全プロジェクトに適用。API キー、テーマ、デフォルトのパーミッション設定など。

.claude/settings.json プロジェクト設定

Git にコミットしてチームで共有。プロジェクト固有のフック、MCP、パーミッション設定。

.claude/settings.local.json ローカル設定

.gitignore に追加。個人のローカル環境固有の設定（ローカル MCP サーバーなど）。

主な設定カテゴリ

{ // パーミッション設定 "permissions": { "allow": ["Edit", "Read", "Bash(npm test)"], "deny": ["Bash(rm -rf *)"] }, // フック設定 "hooks": { /* ... */ }, // MCP サーバー設定 "mcpServers": { /* ... */ }, // モデル設定 "model": "claude-sonnet-4-20250514", // エフォートレベル "effortLevel": "high", // テーマ "theme": "dark", // 自動メモリ "autoMemoryEnabled": true }

設定の優先度

1 Managed（管理者設定） ー 最高優先度
↓
2 CLI フラグ（コマンドライン引数）
↓
3 Local（settings.local.json）
↓
4 Project（settings.json）
↓
5 User（~/.claude/settings.json） ー 最低優先度

まとめと次のステップ

Claude Code のカスタマイズ機能は、プロジェクトの要件に合わせて AI エージェントの振る舞いを細かく制御するための強力なツールセットです。

カスタマイズ機能のまとめ

1 CLAUDE.md ー プロジェクトの基本ルールと記憶を定義
2 .claude/rules/ ー モジュール式のルールで関心事を分離
3 Auto Memory ー Claude が自動的に学習・記憶
4 Skills ー カスタムコマンドで機能を拡張
5 Hooks ー ライフサイクルイベントで自動化
6 MCP ー 外部ツール・サービスとの連携
7 設定ファイル体系 ー 階層的な設定管理で柔軟に制御

確認チェックリスト

  • CLAUDE.md の配置場所と優先度を理解している
  • ルールファイルのパス固有ルール（YAML フロントマター）を使える
  • Auto Memory の仕組みと /memory コマンドを知っている
  • Skills の SKILL.md 構造とフロントマターフィールドを理解している
  • Hooks のイベント種類と終了コードの意味を把握している
  • MCP サーバーの追加・管理方法とスコープの違いを理解している
  • 設定ファイルの優先度（Managed > CLI > Local > Project > User）を覚えている

次のステップ

カスタマイズの基礎を理解したら、実際のプロジェクトで CLAUDE.md を作成し、必要に応じて Skills や Hooks を追加していきましょう。まずは /init コマンドで CLAUDE.md のひな型を生成するところから始めるのがおすすめです。
