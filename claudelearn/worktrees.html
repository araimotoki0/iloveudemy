Worktreesとは何か

参照: Common Workflows / git worktree ドキュメント

Worktree（ワークツリー）は、1つのGitリポジトリに複数の作業ディレクトリを作るGitの機能です。
Claude Codeはこの仕組みを活用して、複数のAIエージェントを同時に走らせる並列処理を実現しています。

身近な例えで理解する

想像してください。あなたは1つのプロジェクトの書類を持っています。

通常のGit（デスクが1つ）

1つのデスクで1人が作業。
別の作業をするには、今の書類を片付けて（git stash）、
別の書類を出す（git checkout）必要がある。

Worktree（デスクが複数）

同じ書類をコピーして複数のデスクに配布。
それぞれのデスクで別の人が同時に作業できる。
終わったら成果を1つにまとめる（マージ）。

Worktreeの仕組みを図で見る

メインのリポジトリ（.git フォルダは1つだけ） my-project/ ← 元の作業ディレクトリ（mainブランチ） ├── src/ ├── package.json └── .git/ ← Gitの本体。ここは共有される my-project-worktree-1/ ← worktree 1（feature-aブランチ） ├── src/ └── package.json ← ファイルは独立したコピー my-project-worktree-2/ ← worktree 2（feature-bブランチ） ├── src/ └── package.json ← こちらも独立したコピー

ポイント

  • Gitの履歴データ（.git）は共有されるので、ディスク容量の無駄がない
  • 各worktreeはそれぞれ別のブランチをチェックアウトしている
  • ファイルの実体は独立しているので、同時に編集しても競合しない

ハンズオン: git worktreeを体験する

まずはGit単体でworktreeの動きを確認しましょう。

1

既存リポジトリでworktreeを作成

git worktree add ../my-feature feature-branch
Preparing worktree (new branch 'feature-branch')
HEAD is now at abc1234 Latest commit message
2

worktreeの一覧を確認

git worktree list
/home/user/my-project abc1234 [main]
/home/user/my-feature abc1234 [feature-branch]
3

不要になったworktreeを削除

git worktree remove ../my-feature

Claude Codeでの活用

Claude Codeでは、上記のworktree操作を自動で行います。
ユーザーが意識しなくても、並列タスクを依頼すると裏でworktreeが作られ、各エージェントが独立した作業場所で動き、完了後に自動的にマージされます。

なぜWorktreesが必要なのか

「AIが賢いなら、1つのディレクトリで全部やればいいのでは？」と思うかもしれません。
しかし、複数のAIが同時にファイルを触ると、必ず問題が起きます。

問題: ファイル競合

同じディレクトリで2つのAIが同時に作業したら... AI-A: src/App.tsx の10行目を編集中... AI-B: src/App.tsx の15行目を編集中... → AI-Aの変更をAI-Bが上書き！ → テストが通らない、ビルドが壊れる... ──────────────────────────────── Worktreeなら... AI-A: worktree-a/src/App.tsx を編集 ← 完全に独立 AI-B: worktree-b/src/App.tsx を編集 ← 完全に独立 → 作業後にGitが賢くマージ → 競合がなければ自動解決！

Worktreeが解決する3つの課題

1. ファイル競合の回避

各AIが独立したディレクトリで作業するため、同じファイルを同時に編集しても衝突しない。

2. 劇的な速度向上

10個のタスクを順番に実行すると10倍の時間がかかる。並列ならほぼ同時に完了。

3. 安全なロールバック

各worktreeは独立したブランチ。1つのタスクが失敗しても、他のタスクに影響しない。

実際の速度比較

「5つのコンポーネントにテストを追加して」 直列実行（worktreeなし）: Component-A ████████░░ 2分 Component-B ░░░░░░░░░░ （待機中...） Component-C ░░░░░░░░░░ （待機中...） Component-D ░░░░░░░░░░ （待機中...） Component-E ░░░░░░░░░░ （待機中...） → 合計: 約10分 並列実行（worktreeあり）: Component-A ████████░░ 2分 Component-B ████████░░ 2分 ← 同時進行！ Component-C ████████░░ 2分 ← 同時進行！ Component-D ████████░░ 2分 ← 同時進行！ Component-E ████████░░ 2分 ← 同時進行！ → 合計: 約2分（5倍速い！）

注意: 並列処理が向かないケース

  • タスク間に依存関係がある場合 ― 「タスクAの結果を使ってタスクBを実行」のような場合は直列の方が安全
  • 同じファイルを必ず編集する場合 ― マージ時に競合が発生しやすい
  • APIレート制限 ― 多数のエージェントを同時に動かすと、APIの制限に達する可能性がある

ハンズオン ― /batch で並列実行してみよう

参照: Common Workflows - Batch Processing

/batch コマンドは、Claude Codeの中で最も簡単に並列処理を体験できる機能です。
複数のタスクを一気に渡すと、それぞれがworktree上で独立して実行されます。

基本的な使い方

1

Claude Codeを起動

claude
2

/batch コマンドで複数タスクを投入

Claude Codeのプロンプトで以下のように入力します。

/batch # 以下のようにタスクを記述します: src/配下のすべてのコンポーネントにユニットテストを追加して。 各コンポーネントに対して、基本的なレンダリングテストと propsのテストを含めてください。
3

Claude Codeがタスクを分割して並列実行

Claude Code が自動的にタスクを分解: Worktree 1: src/components/Header.tsx のテスト作成 Worktree 2: src/components/Footer.tsx のテスト作成 Worktree 3: src/components/Sidebar.tsx のテスト作成 Worktree 4: src/components/Button.tsx のテスト作成 Worktree 5: src/components/Modal.tsx のテスト作成 ↓ すべて同時に実行中... ↓ 完了したworktreeから順にマージ ✓ 5つのテストファイルが作成されました

実践例: よくあるbatchの使い方

例1: 複数ファイルのリファクタリング

/batch 以下のファイルをTypeScriptに変換して: - src/utils/helpers.js - src/utils/validators.js - src/utils/formatters.js それぞれ適切な型定義を追加すること。

例2: ドキュメントの一括生成

/batch src/api/配下のすべてのエンドポイントに対して、 JSDocコメントを追加して。リクエスト/レスポンスの型、 エラーケース、使用例を含めること。

例3: バグ修正の一括対応

/batch 以下のGitHub Issueをそれぞれ修正して: - #123: ログインボタンが二重クリックできる問題 - #124: メール検証の正規表現が不正 - #125: ページネーションの境界値バグ

最大30エージェントの並列処理

batchの上限

  • /batch は最大30個のサブエージェントを同時に起動できます
  • タスクが30個を超える場合は、自動的にキューイングされます
  • 各エージェントはそれぞれ独立したworktreeで動作するため、互いに干渉しません

ハンズオン: 実際にbatchを試す

簡単なプロジェクトで試してみましょう。

1

テスト用プロジェクトを作成

mkdir batch-test && cd batch-test && git init
2

シンプルなファイルを3つ作成

echo "function add(a, b) { return a + b; }" > math.js
echo "function greet(name) { return 'Hello ' + name; }" > greet.js
echo "function isEven(n) { return n % 2 === 0; }" > check.js
git add -A && git commit -m "initial commit"
3

Claude Codeで/batchを実行

/batch 以下の3つのJSファイルそれぞれにJestテストを作成して: - math.js - greet.js - check.js 各テストファイルは *.test.js という命名にすること。

3つのworktreeが作成され、各エージェントが独立してテストを書きます。

4

結果を確認

ls *.test.js
math.test.js greet.test.js check.test.js

3つのテストファイルが並列で作成されていることを確認できます。

batchのコツ

  • タスクは具体的に ― 曖昧な指示だとClaude Codeがうまく分割できないことがある
  • 独立したタスクを渡す ― タスク間に依存関係がないことが並列処理の前提
  • 完了後にテストを実行 ― マージ後にテストを走らせて、全体の整合性を確認しましょう

サブエージェントとWorktree

参照: Sub-agents

/batch 以外にも、Claude Codeのサブエージェントは自動的にworktreeを活用します。
特に isolation: worktree の設定を理解することが重要です。

サブエージェントがworktreeを使う仕組み

メインエージェント（あなたとの会話） │ ├── サブエージェント A ← 別のworktreeで動作 │ └── worktree-a/ で独立して作業 │ ├── サブエージェント B ← 別のworktreeで動作 │ └── worktree-b/ で独立して作業 │ └── サブエージェント C ← 別のworktreeで動作 └── worktree-c/ で独立して作業 → 各サブエージェントの成果を メインエージェントが集約してマージ

isolation: worktree の設定

カスタムスラッシュコマンド（.claude/commands/に配置するもの）で、サブエージェントにworktreeを使わせるかどうかを制御できます。

isolation: worktree ― 独立したworktreeで実行

# .claude/commands/add-tests.md --- isolation: worktree --- 指定されたコンポーネントにユニットテストを追加してください。 テストは同じディレクトリに .test.tsx として作成すること。

isolation: worktree を指定すると、このコマンドは新しいworktreeで独立して実行されます。
メインの作業ディレクトリには影響を与えず、完了後に結果がマージされます。

context: inherit ― 同じディレクトリで実行（デフォルト）

# .claude/commands/review.md --- context: inherit --- 現在のコードをレビューして、改善点を指摘してください。 ファイルの変更は行わないこと。

context: inherit（デフォルト）は、メインエージェントと同じディレクトリで実行されます。
レビューや分析など、ファイルを変更しないタスクに適しています。

ハンズオン: fork付きコマンドを作ってみよう

1

コマンドファイルを作成

mkdir -p .claude/commands

以下の内容で .claude/commands/parallel-refactor.md を作成します:

--- isolation: worktree --- $ARGUMENTSで指定されたファイルをリファクタリングしてください。 - 関数を小さく分割する - 適切な型定義を追加する - コメントを追加する
2

コマンドを実行

Claude Codeのプロンプトで:

/parallel-refactor src/utils/helpers.ts

isolation: worktree により、別のworktreeでリファクタリングが実行されます。
メインの作業を止めることなく、バックグラウンドで処理が進みます。

Worktreeの自動クリーンアップ

知っておきたいこと

  • サブエージェントが作成したworktreeは、タスク完了後に自動的に削除されます
  • 変更内容はマージされてからworktreeが削除されるので、成果物が失われることはありません
  • 万が一クリーンアップが失敗した場合は、git worktree prune で手動クリーンアップできます

ハンズオン: worktreeのクリーンアップを確認

1

batch実行中にworktreeを確認

別のターミナルで以下を実行:

git worktree list
/home/user/my-project abc1234 [main]
/tmp/claude-worktree-abc123-task1 def5678 [claude/batch-task-1]
/tmp/claude-worktree-abc123-task2 ghi9012 [claude/batch-task-2]

実行中はworktreeが存在していることが確認できます。

2

完了後にもう一度確認

git worktree list
/home/user/my-project abc1234 [main]

worktreeが自動的に削除されていることがわかります。

手動でWorktreeを管理する

通常はClaude Codeが自動でworktreeを管理しますが、手動で制御したい場面もあります。
ここでは、手動管理のためのコマンドを紹介します。

/worktree コマンド

Claude Codeのセッション中に、/worktree コマンドでworktreeを管理できます。

/worktree ― worktreeで作業を開始する

# Claude Codeのプロンプトで: /worktree # 新しいworktreeが作成され、そこで作業を開始します # この状態で行った変更は、元のディレクトリには影響しません

実験的な変更を試したいときに便利です。失敗しても元のコードには影響しません。

ハンズオン: worktreeで安全に実験する

1

worktreeに入る

Claude Codeのプロンプトで:

/worktree
2

大胆な変更を試す

このプロジェクトのルーティングを React RouterからTanStack Routerに全面移行して。

worktree内なので、失敗しても元のコードは安全です。

3

結果を確認して判断

うまくいったら:

# worktreeでの変更がコミットされ、メインブランチにマージされます

うまくいかなかったら:

# worktreeが削除され、何事もなかったように元に戻る

claude --worktree オプション

CLIからClaude Codeを起動するときに、最初からworktreeで作業を開始できます。

claude --worktree
Creating new worktree...
Worktree created at /tmp/claude-worktree-xyz789
You are now working in an isolated worktree.

--worktree の使いどころ

  • 大きなリファクタリングを試す前に ― 「この変更で壊れないか不安...」というとき
  • 複数のアプローチを比較するとき ― 別々のworktreeで異なるアプローチを試して、良い方を採用
  • コードレビュー中の実験 ― PRのコードを動かして確認したいとき、本体を汚さずに試せる

ハンズオン: CLIからworktreeを起動

1

worktreeモードで起動

claude --worktree
2

作業を行う

package.jsonの依存関係をすべて最新バージョンに更新して、 テストが通ることを確認して。
3

テストが通ったらマージ

Claude Codeを終了すると、変更をメインブランチにマージするか聞かれます。

worktree管理のまとめ

方法 コマンド 用途
自動（推奨） /batch 複数タスクの並列実行
自動 isolation: worktree サブエージェントの独立実行
手動 /worktree セッション中に安全な実験環境を作る
CLI起動時 claude --worktree 最初からworktreeで作業開始

トラブルシューティング

  • worktreeが残ってしまった場合: git worktree list で確認し、git worktree remove <path> で削除
  • ロックエラーが出る場合: git worktree prune で不要な参照をクリーンアップ
  • マージ競合が発生した場合: Claude Codeが自動で解決を試みますが、手動対応が必要な場合もあります
