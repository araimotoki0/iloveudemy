Agent Teamsとは何か

参照: Agent Teams

Agent Teamsとは、複数のAIエージェントがチームとして協調して作業する仕組みです。
1つのClaude Codeセッションの中で、複数のエージェントがそれぞれ独立したworktree（作業ディレクトリ）を持ち、並列で作業を進めます。

リフォームで例えると

🔨

従来：1人でリフォーム

大工さんが1人で、壁を壊し、電気配線をやり、水道管を繋ぎ、壁紙を貼る。
全部順番にやるので遅い。
これが通常のClaude Code（1エージェント）です。

👷👷‍♀️👷‍♂️

Agent Teams：チームでリフォーム

大工・電気工事士・水道屋さんがそれぞれの持ち場で同時に作業。
お互いの作業が干渉しないように、別々の部屋（worktree）で作業。
最後に結果を合体させる。

技術的な仕組み

// Agent Teamsの動作イメージ メインエージェント（司令塔） │ ├── エージェント A → worktree-a/ → 「認証機能を実装」 │ ├── エージェント B → worktree-b/ → 「APIエンドポイントを追加」 │ └── エージェント C → worktree-c/ → 「テストを書く」 // 各エージェントは独立したgit worktreeで作業 // → ファイルの競合が起きない！ // → 全員が終わったら結果をマージ

git worktreeとは？

1つのgitリポジトリから、複数の作業コピーを作るgitの機能です。 同じリポジトリなのに、別々のフォルダで別々のブランチを同時に編集できます。
Agent Teamsはこの仕組みを使って、エージェント同士がファイルを奪い合わないようにしています。

Agent Teamsが得意なこと

  • 大規模なリファクタリング ― 10個のファイルを同時に修正
  • 複数機能の同時実装 ― フロント・バック・テストを並列で
  • コードベース全体への変更 ― 全コンポーネントにエラーハンドリング追加
  • マイグレーション ― 複数ファイルのAPI呼び出しを一括更新

ハンズオン ― /batch で大規模変更を並列実行

参照: Agent Teams - Batch

/batch コマンドは、Agent Teamsの中核機能です。
大きなタスクを指示すると、AIが自動でサブタスクに分割し、複数エージェントが並列で処理します。

Step 1: /batch コマンドを実行する

Claude Codeのプロンプトで /batch と入力し、やりたいことを伝えます。

/batch 全コンポーネントにエラーハンドリングを追加して

ポイント: 具体的に指示するほど精度が上がります。「try-catchで囲んで、エラーはconsole.errorで出力して」のように詳細を加えましょう。

Step 2: AIがタスクを自動分割する

AIがコードベースを分析し、変更が必要なファイルを特定。タスクをサブタスクに分割します。

AIの自動分割結果: タスク1: src/components/UserList.tsx にエラーハンドリング追加 タスク2: src/components/Dashboard.tsx にエラーハンドリング追加 タスク3: src/components/Settings.tsx にエラーハンドリング追加 タスク4: src/components/Profile.tsx にエラーハンドリング追加 → 4つのエージェントが並列で作業開始！

Step 3: 並列処理の様子を観察する

各エージェントの進捗がリアルタイムで表示されます。

Agent Teams 実行中... [Agent 1] UserList.tsx ████████████░░░░ 75% try-catch追加中 [Agent 2] Dashboard.tsx ██████████████░░ 88% ErrorBoundary実装中 [Agent 3] Settings.tsx ████████████████ 完了! [Agent 4] Profile.tsx ██████░░░░░░░░░░ 38% 分析中

Step 4: 結果のマージと確認

全エージェントの作業が完了すると、結果が自動でマージされます。

git diff --stat
src/components/UserList.tsx | 15 ++++++++++---
src/components/Dashboard.tsx | 22 +++++++++++++-----
src/components/Settings.tsx | 18 +++++++++++----
src/components/Profile.tsx | 12 ++++++++--
4 files changed, 52 insertions(+), 15 deletions(-)

確認のコツ: マージ後は git diff で変更内容を確認し、 テストを実行して問題がないか検証しましょう。

実践的な /batch の使い方いろいろ

テスト一括追加

/batch 全APIルートにユニットテストを追加して

型定義の追加

/batch 全jsファイルをTypeScriptに変換して

ログ追加

/batch 全サービスに構造化ログを追加して

API移行

/batch fetchをaxiosに全部置き換えて

注意点

  • API使用量が増える: エージェントの数だけAPI呼び出しが発生します。Max プランまたはAPIキーが必要です
  • 競合の可能性: 同じファイルを複数エージェントが編集する場合、マージ時に競合が起きることがあります
  • まず小さく試す: 最初は2〜3ファイル程度で試して、結果を確認してから大規模に実行しましょう

Triggers / 定期実行（/loop と /schedule）

参照: Agent Teams / Scheduled Tasks

Agent Teamsには、タスクを定期的に自動実行する仕組みがあります。
人間が毎回指示しなくても、AIが自律的に監視や作業を続けてくれます。

/loop ― 一定間隔で繰り返し実行

/loop は、指定した間隔でコマンドやタスクを繰り返し実行します。
バックグラウンドで動き続け、あなたは別の作業に集中できます。

/loop 5m PRの状態を確認して、新しいPRがあればレビューして
5分ごとにPRの状態をチェックするループを開始しました。
停止するには Ctrl+F を2回押してください。
/loop の書式: /loop [間隔] [実行したいタスク] // 間隔の例: 30s → 30秒ごと 5m → 5分ごと 1h → 1時間ごと 30m → 30分ごと

/schedule ― cron的な定期実行

/schedule は、より細かいスケジュール設定ができる定期実行です。
「毎朝9時に」「毎週月曜に」のようなスケジュールを指定できます。

/schedule "毎朝9時にmainブランチのテストを実行して結果をSlackに投稿して"

実用例

🔍

デプロイ監視

/loop 2m デプロイ状態を確認して、失敗していたら教えて

CI/CDパイプラインの状態を定期チェック。失敗時にすぐ気づける。

📝

PR自動レビュー

/loop 10m 新しいPRがあればコードレビューして

新しいPRが来るたびにAIが自動でレビュー。人間のレビュー前に問題を発見。

🧪

テスト定期実行

/loop 30m テストを実行して、失敗があれば修正して

テストが壊れたらAIが自動で修正を試みる。常にグリーンを維持。

/loop と /schedule の使い分け

  • /loop → 「今から繰り返して」。作業セッション中の一時的な監視に最適
  • /schedule → 「決まった時間に実行して」。恒常的なタスクの自動化に最適

ハンズオン ― /loop でPR監視してみよう

実際に /loop を使って、PRの自動監視をセットアップしてみましょう。
バックグラウンドで動き続けるAIエージェントを体験できます。

Step 1: GitHubリポジトリで実行する

1

まず、GitHubリポジトリのあるディレクトリでClaude Codeを起動します。

cd ~/my-project && claude

Step 2: /loop でPR監視を開始する

2

Claude Codeのプロンプトで以下を入力します。

/loop 5m オープンなPRを確認して、新しいものがあればコードレビューして
ループを開始しました (間隔: 5分)

[1回目の実行 - 14:30:00]
PR #42 "Add user authentication" を確認中...
→ セキュリティ上の問題を1件発見。コメントを投稿しました。

次回実行: 14:35:00

Step 3: バックグラウンドで動き続ける

3

ループが開始されると、エージェントはバックグラウンドで動き続けます。
あなたは別のタスクを続けることができます。

バックグラウンド動作のイメージ: あなた: 別の機能を実装中... 🧑‍💻 裏で動くAgent: 5分ごとにPRチェック 🔄 14:30 → PR確認... 新着なし 14:35 → PR確認... PR #43 発見！レビュー中... 14:40 → PR確認... 新着なし 14:45 → PR確認... PR #44 発見！レビュー中...

Step 4: ループを停止する

4

ループを止めたいときは、以下の方法で停止できます。

Ctrl+F を2回押す

全エージェントを一括停止します。
Ctrl+F 1回目 → 警告表示
Ctrl+F 2回目 → 完全停止

Ctrl+C

現在のタスクを中断します。
ループ自体は次の実行タイミングで再開される場合があります。
確実に止めるなら Ctrl+F 2回。

ハンズオンの注意

  • APIコストに注意: ループが回るたびにAPI呼び出しが発生します。短い間隔（30sなど）は学習目的のみにしましょう
  • GitHubトークンが必要: PR操作にはGitHub CLIの認証が必要です（gh auth login）
  • ネットワーク接続: オフラインでは動作しません

応用: /loop の便利なパターン

ファイル変更の自動テスト

/loop 1m src/ディレクトリの変更を検知して、変更があればテストを実行して

ファイルを保存するたびに自動でテストが走る、ホットリロード的な体験。

ビルドエラーの自動修正

/loop 3m ビルドを実行して、エラーがあれば修正して

ビルドが壊れてもAIが自動で直してくれる。開発中のストレスが激減。

Remote Control（スマホからClaude Codeを操作）

参照: Remote Control

Remote Controlは、ローカルで動いているClaude CodeをWebブラウザから操作できる機能です。
スマホからでも、別のPCからでも、あなたのClaude Codeに指示を出せます。

どういう仕組み？

Remote Controlの流れ: あなたのPC（ローカル） └── Claude Code が動いている │ │ セキュアな接続 │ claude.ai/code（Webブラウザ） └── スマホ / タブレット / 別PC からアクセス └── Claude Codeに指示を送る └── 結果がリアルタイムで表示される // コードはローカルに残る（クラウドにアップロードされない）

ハンズオン: Remote Controlを使ってみる

1

ローカルでClaude Codeを起動

いつも通り、ターミナルでClaude Codeを起動します。

claude
2

Webブラウザでアクセス

スマホやタブレットで以下のURLにアクセスします。

https://claude.ai/code

ログインすると、ローカルで動いているClaude Codeのセッションが表示されます。

3

スマホから指示を送る

Webブラウザ上でプロンプトを入力すると、ローカルのClaude Codeが実行します。

📱

スマホから送信

「テストを全部実行して結果を教えて」

💻

ローカルPCが実行

Claude Codeがテストを実行し、結果をスマホに返す

Remote Controlの活用シーン

🚶

外出中の確認

移動中にスマホでビルド状態やテスト結果を確認。問題があればその場で修正指示。

🛋️

リラックスしながら

ソファでタブレットからコードレビューを依頼。AIが分析してくれるのを待つだけ。

🔄

長時間タスクの監視

大規模なリファクタリングをAgent Teamsで実行中、スマホで進捗を確認。

👥

ペアプログラミング

同僚に画面を見せながら、自分はスマホからClaude Codeに指示を出す。

セキュリティについて

  • コードはローカルに残る: Remote Controlはあくまで「リモコン」。コードがクラウドにアップロードされることはありません
  • 認証が必要: claude.ai にログインしたアカウントからのみアクセスできます
  • 暗号化通信: ローカルPCとWebブラウザ間の通信はセキュアに暗号化されています

制限事項

  • ローカルPCが起動している必要あり: PCがスリープ/シャットダウンすると接続が切れます
  • Claude Codeが起動している必要あり: ターミナルでClaude Codeを閉じると使えません
  • ネットワーク接続: ローカルPCとスマホの両方がインターネットに接続されている必要があります
