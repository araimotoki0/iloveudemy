AIモデルとハーネス ― Claude Codeが「エージェント」になれる理由

Claude Codeを理解するうえで最も重要な概念がハーネスです。

AIモデル単体は「テキストを受け取って返す」だけ

Claude（AIモデル）がやっていることは、突き詰めるとテキストを入力として受け取り、テキストを出力として返すだけです。
ファイルを読むことも、コマンドを実行することも、デプロイすることも、AIモデル単体ではできません。

ハーネスがAIをエージェントにする

ハーネスとは、AIモデルを包み込んで動かすプログラムのことです。Claude Codeの場合、このハーネスが以下を担当しています：

CLAUDE.mdの読み込みとリクエストへの注入
セッション開始時にCLAUDE.mdを読み、メッセージ配列の先頭に差し込む
ツールの実行
AIが「このファイルを読みたい」「このコマンドを実行したい」と返答したら、ハーネスが実際にファイルを読み、コマンドを実行し、結果をAIに返す
会話ループの管理
AIの出力を解析し、ツール呼び出しがあればそれを実行、結果を付けて再度AIにリクエスト ― これを繰り返す（エージェントループ）
権限管理・サンドボックス
危険な操作をブロックしたり、ユーザーに許可を求めたりする
コンテキストウィンドウの管理
会話が長くなったら自動で圧縮（/compact）し、コンテキストの枯渇を防ぐ
【ユーザー】 「ログイン画面を作って」 ↓ ハーネスがCLAUDE.mdを注入してAPIリクエスト送信 【AIモデル（Claude）】 「ファイルを作成します」→ ツール呼び出し: Write("login.tsx", ...) ↓ ハーネスがツールを実行し、結果をAIに返す 【AIモデル（Claude）】 「テストも書きます」→ ツール呼び出し: Write("login.test.tsx", ...) ↓ ハーネスがツールを実行...（繰り返し） 【完了】 ファイルが作成される

つまり

AIモデルは「テキストを返す」だけ。それを受け取って実際にファイルを操作し、コマンドを実行し、結果を返してまたAIに聞く ― このループを回すのがハーネスです。このハーネスがあるからこそ、Claude Codeはエージェントのように振る舞えます。

Claude Codeの仕組みを調べる方法

Claude Codeを使いこなすには、「この機能は内部でどう動いているのか？」を自分で調べられることが重要です。
調べる方法は大きく2つあります。

アプローチ1：ソースコードを読む

Claude CodeはJavaScript（Node.js）で書かれたCLIツールです。インストール方法によって、ソースコードの読みやすさが変わります。

通常インストール（claudeコマンド）
ネイティブバイナリが配布される。
JavaScriptのソースコードがバイナリの中に埋め込まれているため、strings コマンドで文字列を抽出すれば読めるが、周辺のコードを探すのが大変。
npm インストール
ミニファイされたJSファイルがそのまま配置される。
変数名は短縮されているが、テキストファイルなので grep で簡単に検索・調査できる。

なぜネイティブバイナリだと読みにくい？

ネイティブバイナリは、JavaScriptのコードをCPUが直接実行できる機械語に変換（コンパイル）したものです。人間が読めるソースコードの構造は失われています。ただし、文字列リテラル（エラーメッセージやキーワードなど）はバイナリの中にそのまま残っているため、strings コマンドで抽出できます。

npm install は調査専用

npm経由のインストールは通常の利用方法としては非推奨です。あくまで内部実装を調査する目的で使います。

npm install -g @anthropic-ai/claude-code

インストール後、ミニファイされたJSファイルの中を grep で検索できます。ネイティブバイナリの場合は strings コマンドで文字列を抽出してから検索します。

具体例：CLAUDE.mdの注入メカニズムを発見する

「CLAUDE.mdは本当に毎回のリクエストに載るのか？」を調べたケースです。
バイナリから文字列を抽出して、CLAUDE.mdの注入ロジックを特定しました。

grep -boa "Codebase and user" /path/to/claude-binary
69833497:Codebase and user 79063744:Codebase and user

バイトオフセットを使って周辺のコードを抽出すると、注入ロジックが見えます。

dd if=/path/to/claude-binary bs=1 skip=69833400 count=500 2>/dev/null | strings

実際に見つかったソースコード（ミニファイ済み）：

// CLAUDE.mdの内容をラップするテキスト wBK = "Codebase and user instructions are shown below. Be sure to adhere to these instructions. IMPORTANT: These instructions OVERRIDE any default behavior and you MUST follow them exactly as written." // CLAUDE.md読み込み関数（メモ化されている = セッション中1回だけディスクから読む） a3 = Oq(async () => { ... }) // 種類ごとのラベル付け R.type === "Project" ? " (project instructions, checked into the codebase)" : R.type === "Local" ? " (user's private project instructions, not checked in)" : " (user's private global instructions for all projects)"

ソースコードから判明した事実

  • CLAUDE.mdの内容は <system-reminder> タグで囲まれてメッセージ配列の先頭に注入される
  • 読み込み関数はメモ化されている → ディスクから読むのはセッション開始時に1回だけ
  • 一度コンテキストに入ったら、以降のすべてのAPIリクエストに含まれる（会話履歴の一部として）
  • ファイルの種類（Project / Local / User）によって異なるラベルが付与される

アプローチ2：公式ドキュメントを読む + AIに聞く

公式ドキュメントには機能の使い方や設計思想が体系的にまとまっています。

主要な公式ドキュメント

  • 概要 ― Claude Codeとは何か
  • メモリ ― CLAUDE.mdと自動メモリ
  • 設定 ― settings.jsonの構成
  • セッション ― コンテキスト管理と会話の再開
  • フック ― イベント駆動の自動化

ドキュメントだけで解決しない疑問は、Claude自身に聞いてしまうのが最も効率的です。

方法は2つ：

ブラウザ版Claude
claude.ai を開いて質問する。
「Claude CodeのCLAUDE.mdはどう読み込まれる？」のように聞けば、ドキュメントに基づいた回答が得られる。
Claude Code自身に聞く
ターミナルで claude を起動して直接質問する。
Claude Codeは自分自身の仕組みについても回答できる。

2つのアプローチを組み合わせる

公式ドキュメントで全体像を把握し、ソースコード調査で実装の詳細を確認する。疑問があればClaude自身に聞く。この3つを組み合わせることで、ドキュメントに書かれていない実装の詳細まで正確に把握できます。
次のセクションでは、この方法で得た知識をもとにCLAUDE.mdとメモリの仕組みを詳しく解説します。

CLAUDE.md と自動メモリ ― 深掘り

参照: Claude があなたのプロジェクトを記憶する方法

ここからは、公式ドキュメントとソースコード調査で明らかになったCLAUDE.mdと自動メモリの仕組みを具体的に見ていきます。
Claude Codeがセッション間で知識を保持するメカニズムは2つあります。
CLAUDE.md（あなたが書く指示）と自動メモリ（Claudeが自分で書くメモ）です。

CLAUDE.md と自動メモリの比較

CLAUDE.md 自動メモリ
誰が作るか あなた Claude
中身 指示とルール 学習とパターン
共有 Gitでチーム共有可能 マシンローカルのみ
管理のしやすさ 自分で書くから見通しが良い Claudeが勝手に保存するため把握しにくい
読み込み セッション開始時に全文読み込み MEMORY.mdの先頭200行のみ

使い分け：確実に守らせたいルール → CLAUDE.md。Claudeが作業中に自然に学ぶ情報（ビルドコマンド、デバッグの知見など）→ 自動メモリに任せる。

CLAUDE.md ファイルの種類と配置場所

CLAUDE.mdは配置場所によってスコープが変わります。

~/CLAUDE.md（ユーザー指示）
全プロジェクト共通の個人設定。「日本語で返答して」「コードスタイルの好み」など。あなただけに適用。
プロジェクト/CLAUDE.md（プロジェクト指示）
プロジェクト固有のルール。技術スタック、コーディング規約、ワークフロー。Gitでチーム全員と共有。
プロジェクト/CLAUDE.local.md（ローカル指示）
プロジェクト固有だがGitにコミットしない個人設定。.gitignore に入れて使う。自分だけのデバッグ設定や作業メモに。
/Library/Application Support/ClaudeCode/CLAUDE.md（管理ポリシー）
IT/DevOpsが管理する組織全体の指示。セキュリティポリシーやコンプライアンス要件など。個別に除外不可。

読み込みの仕組み

Claude Codeは起動時に、現在のディレクトリからディレクトリツリーを上方向に辿って各階層のCLAUDE.mdを読み込みます。
例えば foo/bar/ で起動すると、foo/bar/CLAUDE.md と foo/CLAUDE.md の両方が読み込まれます。

サブディレクトリのCLAUDE.mdは起動時ではなく、そのディレクトリ内のファイルを読んだときにオンデマンドで読み込まれます。

効果的なCLAUDE.mdの書き方

CLAUDE.mdは「コンテキスト」であり「強制設定」ではありません。具体的で簡潔な指示ほど、Claudeが従う可能性が高くなります。

悪い例 良い例
コードを適切にフォーマットする 2スペースのインデントを使用する
変更をテストする コミット前に npm test を実行する
ファイルを整理しておく APIハンドラーは src/api/handlers/ に配置する

サイズに注意

CLAUDE.mdは200行以下が目安です。長すぎるとコンテキストを消費し、遵守率が下がります。大きくなったら .claude/rules/ に分割しましょう。

大規模プロジェクト：.claude/rules/ でルールを分割

指示が増えてきたら、.claude/rules/ ディレクトリにトピックごとのファイルを置けます。

your-project/ ├── .claude/ │ ├── CLAUDE.md # メインの指示 │ └── rules/ │ ├── code-style.md # コードスタイル │ ├── testing.md # テスト規約 │ └── security.md # セキュリティ要件

さらに、特定のファイルにだけ適用されるルールも書けます。YAMLフロントマターで対象パスを指定します。

# .claude/rules/api-rules.md --- paths: - "src/api/**/*.ts" --- - すべてのAPIエンドポイントに入力バリデーションを含める - 標準エラーレスポンス形式を使用する - OpenAPIドキュメンテーションコメントを含める

このルールは src/api/ 配下のTypeScriptファイルを読んだときだけ読み込まれるため、関係ないファイルの作業時にコンテキストを消費しません。

個人的なルールは ~/.claude/rules/ に置けば、すべてのプロジェクトに適用されます。

モノレポ：不要なCLAUDE.mdを除外する

大規模なモノレポでは、他チームのCLAUDE.mdが読み込まれてしまうことがあります。claudeMdExcludes 設定で除外できます。

// .claude/settings.local.json { "claudeMdExcludes": [ "**/monorepo/CLAUDE.md", "/home/user/monorepo/other-team/.claude/rules/**" ] }

管理ポリシーのCLAUDE.mdは除外できません（組織全体のルールは常に適用されます）。

外部ファイルのインポート

CLAUDE.mdの中で @パス と書くと、そのファイルの内容を取り込めます。

# CLAUDE.md プロジェクト概要は @README を参照 npmコマンドは @package.json を参照 # 個人的な好み（Gitに入らない） - @~/.claude/my-project-instructions.md

自動メモリの仕組み

自動メモリはClaude Codeが作業中に自分でメモを書く機能です。ビルドコマンド、デバッグの知見、コードスタイルの好みなどを自動で保存します。

メモリファイルは ~/.claude/projects/<project>/memory/ に保存されます。

~/.claude/projects/<project>/memory/ ├── MEMORY.md # インデックス（先頭200行がセッション開始時に読み込まれる） ├── debugging.md # デバッグパターン ├── api-conventions.md # API設計の決定 └── ...

メモリの内容はすべてプレーンなMarkdownなので、いつでも確認・編集・削除できます。

自動メモリの有効/無効

自動メモリはデフォルトで有効です。無効にしたい場合は /memory コマンドからトグルするか、設定ファイルで無効にできます。

// プロジェクト設定で無効化 { "autoMemoryEnabled": false }

環境変数 CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 でも無効化できます。

/memory
読み込まれているCLAUDE.mdとメモリファイルの一覧表示、自動メモリのオン/オフ切り替え

ハンズオン：/init でCLAUDE.mdを自動生成してみよう

プロジェクトのCLAUDE.mdを一から書くのは大変です。/init コマンドを使えば、Claudeがコードベースを分析して自動生成してくれます。

/init
コードベースを分析し、ビルドコマンド・テスト指示・プロジェクト規約を含むCLAUDE.mdを生成

すでにCLAUDE.mdがある場合は上書きせず、改善を提案してくれます。生成されたものをベースに、自分でカスタマイズしていきましょう。

ハンズオン：/memory でメモリを確認してみよう

Claudeが何を記憶しているか気になったら、/memory で確認してみましょう。

/memory

読み込まれているCLAUDE.mdの一覧と、自動メモリの内容を確認できます。不要なメモリは編集・削除できます。

トラブルシューティング

  • CLAUDE.mdに従ってくれない？ → /memory でファイルが読み込まれているか確認。指示をより具体的に。矛盾する指示がないかチェック。
  • CLAUDE.mdが大きくなりすぎた？ → 200行を超えたら .claude/rules/ や @インポート で分割。
  • /compact 後に指示が消えた？ → CLAUDE.mdに書いた指示は圧縮後も再読み込みされます。消えたのは会話内で伝えただけの指示です。CLAUDE.mdに追加しましょう。
