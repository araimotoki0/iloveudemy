AIモデルとハーネス ― Claude Codeが「エージェント」になれる理由

Claude Codeを理解するうえで最も重要な概念がハーネスです。

AIモデル単体は「テキストを受け取って返す」だけ

Claude（AIモデル）がやっていることは、突き詰めるとテキストを入力として受け取り、テキストを出力として返すだけです。
ファイルを読むことも、コマンドを実行することも、デプロイすることも、AIモデル単体ではできません。

ハーネス = AIを包む「もう一つのプログラム」

ハーネスとは、AIモデルを包み込んで動かす別のプログラムのことです。
Claude Codeの場合、ハーネスの正体はTypeScriptで書かれた約51万行のCLIプログラムです。あなたのPCにインストールされて動いています。

💡 実際に確認できます

ターミナルで which claude と打つと、Claude Codeの実行ファイルのパスが表示されます。それがハーネスです。AIモデル本体はAnthropicのサーバー上にあり、ハーネスがAPIを通じてやり取りしています。

ハーネスが担当していること

システムプロンプトの組み立て
CLAUDE.mdの読み込み、ツール一覧の定義、現在のgit状態など ― AIに渡す「指示書」をハーネスが毎回組み立てている
ツールの実行
AIが「このファイルを読みたい」「このコマンドを実行したい」とテキストで返答 → ハーネスが実際にファイルを読み、コマンドを実行し、結果をAIに返す
エージェントループの制御
AIの出力を解析 → ツール呼び出しがあれば実行 → 結果を付けて再度AIにリクエスト ― この繰り返しを管理する
権限管理・サンドボックス
危険な操作をブロックしたり、ユーザーに許可を求めたりする
コンテキストウィンドウの管理
会話が長くなったら自動で圧縮（/compact）し、コンテキストの枯渇を防ぐ
【ユーザー】 「ログイン画面を作って」 ↓ ハーネス（TypeScript）がCLAUDE.mdやツール定義を組み立て、Claude APIに送信 【AIモデル（Claude） ― Anthropicのサーバー上】 「ファイルを作成します」→ テキストで返答: Write("login.tsx", ...) ↓ ハーネスがこのテキストを解析し、実際にファイルを書き込み、結果をAPIに返す 【AIモデル（Claude）】 「テストも書きます」→ テキストで返答: Write("login.test.tsx", ...) ↓ ハーネスが実行...（ツール呼び出しがなくなるまで繰り返し） 【完了】 ファイルが作成される

つまり

AIモデルは「テキストを返す」だけ。それを受け取って実際にファイルを操作し、コマンドを実行し、結果を返してまたAIに聞く ― このループを回すのがハーネスです。このハーネスがあるからこそ、Claude Codeはエージェントのように振る舞えます。

🔥 ハーネス戦争 ― 各社が本当に獲りに行っているもの

ここ、めちゃくちゃ大事な話です。
2025年〜2026年にかけて、AIの世界で最も激しい競争が起きている場所がどこかわかりますか？
モデルの性能？ ― もちろん重要。でも、本当の主戦場はそこじゃない。
「ハーネス」の覇権争いです。

なぜハーネスが主戦場なのか

前のセクションで見たとおり、AIモデルは「テキストを返すだけ」です。同じClaudeモデル、同じGPTモデルを使っても、ハーネスが違えば、まったく別の製品になります。

つまり、AIの世界で起きていることは ―
「AIモデルの頭脳」は共有資源になりつつあり、「その頭脳をどう使うか」の設計で勝負が決まるということ。
だから各社が死に物狂いでハーネスを作っています。

各社が何を狙っているのか

Anthropic ― Claude Code
狙い：「開発者のワークフロー全体を握る」
ターミナル・VS Code・デスクトップアプリ・ブラウザ・Slack ― あらゆる場所にClaude Codeを埋め込む。自社モデル × 自社ハーネスで垂直統合。CLAUDE.md・MCP・Hooks・サブエージェント・スキルと、ハーネスの機能を猛烈に拡張中。
OpenAI ― Codex + OpenClawの人材獲得
狙い：「汎用AIエージェント基盤を制する」
Codex（クラウド型コーディングエージェント）を投入しつつ、GitHub史上最速で24.7万スターを獲得したOpenClawの開発者Peter Steinbergerを採用。コーディング以外のあらゆる領域に広げようとしている。
Cursor ― IDE統合型 + 自社モデル開発
狙い：「エディタそのものをAIエージェントにする」
VS Codeをフォークし、エディタ自体にハーネスを組み込んだ。当初は外部モデル（Claude・GPT等）を切り替えて使う「ハーネス専業」だったが、Composer 2で自社モデル開発に踏み込んだ。OSSのKimi K2.5をベースに、Cursor環境での大規模強化学習を実施。ハーネス企業がモデルも手に入れた ― 戦局が変わりつつある。

全員がやっていることは同じ

見た目も操作方法もまるで違う。しかし、各社のハーネスがやっていることの本質は全部同じです。
「AIモデルに、どれだけ適切なコンテキスト（情報）を渡せるか」 ― これが勝負。

Claude Code → CLAUDE.md、git状態、ファイル内容をコンテキストとして自動で組み立てる
Cursor → プロジェクト全体をインデックス化し、.cursorrules でルールをコンテキストに注入
OpenClaw → ローカルにメモリを蓄積し、24時間365日コンテキストを持続

この講座で学ぶことの意味

Claude Codeを学ぶということは、各社が競争している「ハーネス」の設計思想を、手を動かしながら理解するということです。ツールが変わっても「ハーネスがAIに何を渡しているか」を理解している人はすぐに適応できます。

💬 実話：ソースコード流出事件 ― ハーネスの中身が丸見えになった日

何が起きたのか

2026年3月31日、Anthropicがnpmパッケージ @anthropic-ai/claude-code v2.1.88 を公開した際、ソースマップファイル（.map）を誤って同梱してしまいました。

ソースマップとは、本来開発者がデバッグ用に使う内部ファイルです。圧縮・難読化されたコードと元のソースコードの対応関係を記録するためのもので、その中身には元のTypeScriptソースコード本体が文字列として丸ごと埋め込まれています（sourcesContent フィールド）。つまり「復元した」のではなく、ソースコードそのものがファイルの中にそのまま入っていたということです。

数時間のうちにGitHubにミラーされ、84,000スター・82,000フォークに到達。韓国の開発者がPythonで全体を書き直すなど、世界中で解析が始まりました。

漏れたもの・漏れなかったもの

漏れたもの（ハーネス側）
・システムプロンプトの全文
・ツール実行の実装コード
・エージェントループの制御ロジック
・未公開のフィーチャーフラグ44個
・未発表機能（KAIROS、Buddy等）
漏れなかったもの（モデル側）
・AIモデルの重み・学習データ
・ユーザーの会話履歴
・APIキー・認証情報
・ユーザーの個人データ

この事件は「AIモデルとハーネスは完全に別物」であることを図らずも証明しました。ハーネス（TypeScriptプログラム）が丸ごと漏れても、AIモデル本体には一切影響がなかったのです。

逆に言えば、あなたのPCで動いているClaude Codeの大部分は「普通のプログラム」です。AIの魔法に見える動作の裏側には、ファイルを読み書きし、コマンドを実行し、APIを叩く ― そういう地道なコードの積み重ねがあります。

フォークされたリポジトリは今もGitHub上に残っており、ハーネスの中身を自分の目で確認できます。「ハーネスとは何か」を理解するのに最も直接的な方法です。

①システムプロンプト ― Claude Codeが毎回AIに渡している「指示書」

あなたが claude コマンドを打つ前に、ハーネスがすでにAIに渡している長大な指示書があります。「あなたはClaude Codeというツールです」「ツールはこう使ってください」「コードを書くときのルール」「禁止事項」など、モデルの振る舞いを決めている文章の全文です。

Piebald-AIの system-prompts/ ディレクトリには、システムプロンプトが機能ごとに80個以上のMarkdownファイルに分割されて置かれています。代表的なものを開いてみてください：

📄 system-prompt-communication-style.md
「Claudeはどうユーザーに話しかけるか」の指示。「ツール実行の前に1文で意図を述べよ」「コード内のコメントは原則書くな」など
📄 system-prompt-doing-tasks.md
タスク実行の基本方針。「既存ファイル編集を優先」「セキュリティ脆弱性に注意」「ドキュメントを勝手に作るな」など
📄 system-prompt-tool-usage-policy.md
どのツールをいつ使うべきかのポリシー
📄 system-prompt-auto-mode.md
自動モード（ユーザー確認を省略するモード）の挙動

なぜ読む価値がある？ CLAUDE.mdを書くときのお手本として使えます。「AnthropicはClaudeにどういう指示を与えているのか」を見ると、自分のプロジェクトにどう指示を書けば効くかがわかります。「なぜClaudeはこう返してくるのか」の謎もここで解けます。

②ツール定義 ― AIが使える「道具」の説明書

Claude Codeは50個以上の組み込みツール（Read・Write・Edit・Bash・Grep・Glob・WebFetch・Task等）を持っています。ハーネスは各ツールに対して「この道具はこう使え」という説明文をAIに渡しています。AIはその説明を読んで、どの道具をどう呼び出すかを判断しています。

例えば、ファイルを読むツール Read（内部名: ReadFile）の説明文は tool-description-readfile.md に入っています。中身はこんな感じです（抜粋）：

Reads a file from the local filesystem. You can access any file directly by using this tool. Assume this tool is able to read all files on the machine. Usage: - The file_path parameter must be an absolute path, not a relative path - By default, it reads up to ${MAX_READ_LINES} lines from the beginning - This tool allows Claude Code to read images (eg PNG, JPG, etc) - This tool can read PDF files (.pdf). For large PDFs (more than 10 pages), you MUST provide the pages parameter to read specific page ranges - This tool can read Jupyter notebooks (.ipynb files) - This tool can only read files, not directories. To read a directory, use an ls command via the Bash tool. - If you read a file that exists but has empty contents you will receive a system reminder warning in place of file contents.

これを読むだけで、普段なんとなく体験していた挙動の理由がわかります：

  • なぜReadツールは絶対パス必須なのか → 明文化されている
  • PDFで10ページを超えると pages 指定必須 → ここに書かれている
  • ディレクトリを読もうとするとBashのlsを案内される → この指示のため

他のツールも同じ形式で読めます：

📄 tool-description-edit.md
Editツール。「readしないとeditできない」「old_stringが一意でないと失敗する」などの制約の理由
📄 tool-description-bash.md
Bashツール。「findではなくGlobを使え」「catではなくReadを使え」などツール選択の優先順位
📄 tool-description-webfetch.md
WebFetchツール。「GitHubのURLはgh CLIを優先」などの例外ルール
📄 system-prompts/ ディレクトリ全体を見る
50個以上のツール定義が並んでいる

なぜ読む価値がある？ 「普段なんとなく感じていたClaude Codeの挙動の理由」が全部明文化されています。さらに、自分でMCPツールやスキルを作るとき、Anthropicのツール説明文の書き方そのものがお手本になります。

AIの賢さはモデル側の話。ハーネス側は「システムプロンプトを組み立てる」「ツールを実行する」「ループを回す」という地道なプログラミングの積み重ねです。この理解があると、Claude Codeの挙動に困ったとき「ハーネス側の問題か、モデル側の問題か」を切り分けて考えられるようになります。

コンテキストエンジニアリング ― AIの性能を決めるのは「何を渡すか」

AIモデルの知能は急速に向上し、もはや「頭が良いかどうか」では差がつきにくくなりました。
では、何が差を生むのか？ ― それがコンテキスト（AIに渡す情報）です。

そもそもコンテキストとは？

AIモデルは毎回「渡されたテキスト」だけを見て回答します。過去の会話、あなたのプロジェクト構成、コーディング規約 ― 渡されなければ、存在しないのと同じです。

コンテキストなし：「認証機能を作って」 → AIは一般的なログインフォームを作る（あなたのプロジェクトのことは何も知らない） コンテキストあり：「認証機能を作って」＋ CLAUDE.md ＋ 既存コード ＋ 使用ライブラリ情報 → AIはあなたのプロジェクトに合った実装を作る

この「AIにどんな情報を、どのタイミングで、どう渡すか」を設計することをコンテキストエンジニアリングと呼びます。

あなたにとっての意味

モデルの知能は各社が競争で上げてくれます。あなたがコントロールできるのはコンテキスト側です。

CLAUDE.mdを丁寧に書く。プロジェクト構成を整理する。MCPで必要な外部データに接続する ― これらは全てコンテキストエンジニアリングです。

なぜ「データの蓄積」がこれからの最重要課題なのか

コンテキストエンジニアリングが重要だとわかったところで、次の疑問が浮かびます。
「渡すべきコンテキスト」は、どこから来るのか？

コンテキストの材料は「あなたのデータ」

AIに渡すコンテキストは、空から降ってくるわけではありません。あなたが持っているデータそのものです。

個人の場合
・過去に書いたコード、設計ドキュメント
・プロジェクトの構成やコーディング規約（CLAUDE.md）
・議事録、仕様書、やり取りの履歴
・自分の得意分野や好みの技術スタック
会社・チームの場合
・社内ナレッジベース、マニュアル、FAQ
・顧客データ、問い合わせ履歴、対応ログ
・業務フロー、意思決定の記録
・過去のプロジェクトの成功・失敗事例

なぜ今すぐ始めるべきか？ ― 3つの理由

理由1：データがなければ、AIは「汎用」止まり

どんなに賢いAIモデルでも、あなたの会社のことを知らなければ誰にでも出せる一般的な回答しか返せません。

データなし：「うちの売上が落ちている原因を分析して」 → 「一般的に売上低下の原因は、市場環境、競合、価格設定…」（教科書レベル） データあり：同じ質問 ＋ 過去3年の売上データ ＋ 顧客アンケート ＋ 競合情報 → 「Q3に離脱した顧客の78%がこの機能の欠如を指摘。競合Xが同時期にこの機能をリリース…」（具体的で行動に直結）

データの差が、AIのアウトプットの差に直結する。モデルの知能が同じでも、渡すデータで結果は天と地ほど変わります。

理由2：データは一朝一夕には作れない

AIモデルは課金すれば誰でも使えます。しかし、あなたの会社の10年分の業務データやあなた個人の経験から生まれたナレッジは、今日始めても明日手に入るものではありません。

つまりデータは時間をかけて蓄積するしかないものです。だからこそ、早く始めた人・会社が有利になります。

理由3：データが「競争優位」になる時代

AIモデルの知能は、OpenAI・Anthropic・Googleが競争で上げてくれます。モデルは全員が同じものを使える。差がつくのは「何を渡すか」です。

具体的に何をすればいいか

個人：AIに渡せる形でナレッジを残す
CLAUDE.mdにプロジェクトのルールを書く。設計判断の理由をドキュメントに残す。「なぜそうしたか」を記録する習慣をつける
会社：社内データを構造化・蓄積する
業務マニュアル、FAQ、対応履歴、ナレッジベースを整備する。暗黙知を言語化してデータにする。「頭の中にしかない情報」をなくす
共通：「AIに渡すつもり」でデータを作る
議事録を取る、仕様書を書く、振り返りを記録する ― 今までは「人が読むため」だったが、これからは「AIにコンテキストとして渡すため」にもデータを作る

まとめ

AIの知能はお金で買える。しかしあなたのデータは、あなたしか持っていない。データを意識的に蓄積し、AIに渡せる形で整理すること ― それが個人にとっても会社にとっても、AI時代の最大の競争優位になります。

あなたのアプリも「Agent Wrapper」になる

ハーネス戦争はClaude CodeやCursorの話 ― 自分には関係ない？
いいえ。あなたが作るアプリも同じ構造になっていくという話です。

アプリの役割が変わる

従来のアプリ
ユーザー ↓ 操作 アプリ（機能・ロジック） ↓ 処理 結果を表示
Agent Wrapper（= ハーネス）
ユーザー ↓ 指示 アプリ（コンテキストを組み立てる） ↓ AIに渡す AIモデル（処理） ↓ 結果 アプリが整形して表示

Claude Codeのハーネスがやっていたこと ― コンテキストを組み立ててAIに渡す ― とまったく同じ構造です。アプリそのものが「ハーネス」になる。これをAgent Wrapperと呼びます。

用語について ― 「wrapper」の価値逆転ストーリー

実は「Agent Wrapper」は世界標準の用語ではありません。英語圏でよく使われるのは「LLM wrapper」「harness（ハーネス）」「application layer（アプリケーション層）」などです。ただし意味はどれも同じ ― 「AIモデルを包んでコンテキストを渡す仕組み」のことです。

面白いのは、この「wrapper」という言葉の価値が完全に逆転したことです。

2023〜2024年頃
「これ、ただのGPT wrapperじゃん」
＝ バカにする言葉だった。「APIを呼ぶだけなら誰でもできる、すぐOpenAIに潰される」と言われていた
2025〜2026年
「本気でwrapperを作ることこそ勝ち筋」
＝ Cursor（評価額90億ドル）、Perplexity（90億ドル）、Harvey（50億ドル）、Sierra（45億ドル）、Glean（70億ドル）が証明してしまった

モデル企業（Anthropic・OpenAI）は全分野で最高の製品を作れません。その隙間で「特定ドメインのコンテキストとUXを握った企業」が数千億円〜兆円規模の価値を作っています。「ただのwrapper」と呼ばれていた設計パターンが、AI時代の主戦場そのものになったということです。

あらゆる業界で起きている

カスタマーサポート
AIモデルは汎用。しかしその会社のFAQ・過去の対応履歴・製品情報をコンテキストとして渡すことで、その会社専用のサポートAIになる
営業・マーケティング
顧客データ、商談履歴、市場データ ― 蓄積されたデータをコンテキストとして渡すことで、汎用AIが「自社専属の分析チーム」になる
あなたが作るアプリ
「すごい機能を実装する」時代から、「適切なデータを集めてAIに渡す仕組みを作る」時代へ。MCP、RAG、メモリ設計 ― これらがアプリの核心になる

だからClaude Codeで学ぶことに意味がある

Claude Codeは、最も完成度の高いAgent Wrapperの一つです。この講座でCLAUDE.md・MCP・Hooksを設計する経験は、自分のアプリをAgent Wrapperとして設計する力にそのままつながります。

Claude Codeの仕組みを調べる方法

Claude Codeを使いこなすには、「この機能は内部でどう動いているのか？」を自分で調べられることが重要です。
調べる方法は大きく2つあります。

アプローチ1：流出したソースコードをClaude Codeに読ませる【ハンズオン】

前のセクションで紹介したソースコード流出事件のおかげで、今やClaude Codeの全TypeScriptソース（約51万行）がGitHub上に残っています。これを使わない手はありません。

やり方は単純です：流出コードをcloneして、Claude Code自身に読ませて説明させる。すると「この機能は内部でどう実装されているのか？」という疑問に、実際のソースコードを根拠にした回答が返ってきます。

⚠️ 必ず守るルール（ここだけ注意）

  • npm install は絶対にしない。package.json の postinstall スクリプトが自動実行されて危険
  • node cli.js などで実行しない。読むだけにする
  • VS Codeで開く場合は「ワークスペースを信頼しますか？」で「信頼しない」を選ぶ

この3つを守れば、cloneして読むだけ・Claude Codeに読ませるだけは安全です（ファイルとして存在するだけで実行されることはありません）。

💡 なぜこれが強力なのか

「Claude Code自身がClaude Codeの仕様書」になります。AIに質問すると、通常は訓練データに基づいた一般的な回答が返ってきますが、この方法では実物のコードを読んだうえで答えてくれます。

CLAUDE.md と自動メモリ ― 深掘り

参照: Claude があなたのプロジェクトを記憶する方法

ここからは、公式ドキュメントとソースコード調査で明らかになったCLAUDE.mdと自動メモリの仕組みを具体的に見ていきます。
Claude Codeがセッション間で知識を保持するメカニズムは2つあります。
CLAUDE.md（あなたが書く指示）と自動メモリ（Claudeが自分で書くメモ）です。

CLAUDE.md と自動メモリの比較

CLAUDE.md 自動メモリ
誰が作るか あなた Claude
中身 指示とルール 学習とパターン
共有 Gitでチーム共有可能 マシンローカルのみ
管理のしやすさ 自分で書くから見通しが良い Claudeが勝手に保存するため把握しにくい
読み込み セッション開始時に全文読み込み MEMORY.mdの先頭200行のみ

使い分け：確実に守らせたいルール → CLAUDE.md。Claudeが作業中に自然に学ぶ情報（ビルドコマンド、デバッグの知見など）→ 自動メモリに任せる。

CLAUDE.md ファイルの種類と配置場所

CLAUDE.mdは配置場所によってスコープが変わります。

~/CLAUDE.md（ユーザー指示）
全プロジェクト共通の個人設定。「日本語で返答して」「コードスタイルの好み」など。あなただけに適用。
プロジェクト/CLAUDE.md（プロジェクト指示）
プロジェクト固有のルール。技術スタック、コーディング規約、ワークフロー。Gitでチーム全員と共有。
プロジェクト/CLAUDE.local.md（ローカル指示）
プロジェクト固有だがGitにコミットしない個人設定。.gitignore に入れて使う。自分だけのデバッグ設定や作業メモに。
/Library/Application Support/ClaudeCode/CLAUDE.md（管理ポリシー）
IT/DevOpsが管理する組織全体の指示。セキュリティポリシーやコンプライアンス要件など。個別に除外不可。

読み込みの仕組み

Claude Codeは起動時に、現在のディレクトリからディレクトリツリーを上方向に辿って各階層のCLAUDE.mdを読み込みます。
例えば foo/bar/ で起動すると、foo/bar/CLAUDE.md と foo/CLAUDE.md の両方が読み込まれます。

サブディレクトリのCLAUDE.mdは起動時ではなく、そのディレクトリ内のファイルを読んだときにオンデマンドで読み込まれます。

効果的なCLAUDE.mdの書き方

CLAUDE.mdは「コンテキスト」であり「強制設定」ではありません。具体的で簡潔な指示ほど、Claudeが従う可能性が高くなります。

悪い例 良い例
コードを適切にフォーマットする 2スペースのインデントを使用する
変更をテストする コミット前に npm test を実行する
ファイルを整理しておく APIハンドラーは src/api/handlers/ に配置する

サイズに注意

CLAUDE.mdは200行以下が目安です。長すぎるとコンテキストを消費し、遵守率が下がります。大きくなったら .claude/rules/ に分割しましょう。

大規模プロジェクト：.claude/rules/ でルールを分割

指示が増えてきたら、.claude/rules/ ディレクトリにトピックごとのファイルを置けます。

your-project/ ├── .claude/ │ ├── CLAUDE.md # メインの指示 │ └── rules/ │ ├── code-style.md # コードスタイル │ ├── testing.md # テスト規約 │ └── security.md # セキュリティ要件

さらに、特定のファイルにだけ適用されるルールも書けます。YAMLフロントマターで対象パスを指定します。

# .claude/rules/api-rules.md --- paths: - "src/api/**/*.ts" --- - すべてのAPIエンドポイントに入力バリデーションを含める - 標準エラーレスポンス形式を使用する - OpenAPIドキュメンテーションコメントを含める

このルールは src/api/ 配下のTypeScriptファイルを読んだときだけ読み込まれるため、関係ないファイルの作業時にコンテキストを消費しません。

個人的なルールは ~/.claude/rules/ に置けば、すべてのプロジェクトに適用されます。

モノレポ：不要なCLAUDE.mdを除外する

大規模なモノレポでは、他チームのCLAUDE.mdが読み込まれてしまうことがあります。claudeMdExcludes 設定で除外できます。

// .claude/settings.local.json { "claudeMdExcludes": [ "**/monorepo/CLAUDE.md", "/home/user/monorepo/other-team/.claude/rules/**" ] }

管理ポリシーのCLAUDE.mdは除外できません（組織全体のルールは常に適用されます）。

外部ファイルのインポート

CLAUDE.mdの中で @パス と書くと、そのファイルの内容を取り込めます。

# CLAUDE.md プロジェクト概要は @README を参照 npmコマンドは @package.json を参照 # 個人的な好み（Gitに入らない） - @~/.claude/my-project-instructions.md

自動メモリの仕組み

自動メモリはClaude Codeが作業中に自分でメモを書く機能です。ビルドコマンド、デバッグの知見、コードスタイルの好みなどを自動で保存します。

メモリファイルは ~/.claude/projects/<project>/memory/ に保存されます。

~/.claude/projects/<project>/memory/ ├── MEMORY.md # インデックス（先頭200行がセッション開始時に読み込まれる） ├── debugging.md # デバッグパターン ├── api-conventions.md # API設計の決定 └── ...

メモリの内容はすべてプレーンなMarkdownなので、いつでも確認・編集・削除できます。

自動メモリの有効/無効

自動メモリはデフォルトで有効です。無効にしたい場合は /memory コマンドからトグルするか、設定ファイルで無効にできます。

// プロジェクト設定で無効化 { "autoMemoryEnabled": false }

環境変数 CLAUDE_CODE_DISABLE_AUTO_MEMORY=1 でも無効化できます。

/memory
読み込まれているCLAUDE.mdとメモリファイルの一覧表示、自動メモリのオン/オフ切り替え

ハンズオン：/init でCLAUDE.mdを自動生成してみよう

プロジェクトのCLAUDE.mdを一から書くのは大変です。/init コマンドを使えば、Claudeがコードベースを分析して自動生成してくれます。

/init
コードベースを分析し、ビルドコマンド・テスト指示・プロジェクト規約を含むCLAUDE.mdを生成

すでにCLAUDE.mdがある場合は上書きせず、改善を提案してくれます。生成されたものをベースに、自分でカスタマイズしていきましょう。

ハンズオン：/memory でメモリを確認してみよう

Claudeが何を記憶しているか気になったら、/memory で確認してみましょう。

/memory

読み込まれているCLAUDE.mdの一覧と、自動メモリの内容を確認できます。不要なメモリは編集・削除できます。

トラブルシューティング

  • CLAUDE.mdに従ってくれない？ → /memory でファイルが読み込まれているか確認。指示をより具体的に。矛盾する指示がないかチェック。
  • CLAUDE.mdが大きくなりすぎた？ → 200行を超えたら .claude/rules/ や @インポート で分割。
  • /compact 後に指示が消えた？ → CLAUDE.mdに書いた指示は圧縮後も再読み込みされます。消えたのは会話内で伝えただけの指示です。CLAUDE.mdに追加しましょう。
