🔗 イメージとコンテナの関係

イメージとコンテナの関係を理解することは、Dockerを使いこなす上で非常に重要です。 ここでは、両者の関係性を実践的に学びます。

基本的な関係

nginx:alpine

ポート: 8080

💡 重要なポイント

  • 1つのイメージから、何個でもコンテナを作成できる
  • 各コンテナは独立しており、互いに影響しない
  • イメージは変更されないため、何度でも同じ状態のコンテナを作れる

🎯 ハンズオン: 1つのイメージから複数のコンテナを作成

実際に1つのイメージから複数のコンテナを起動してみましょう。

11つ目のコンテナを起動

docker run -d -p 8080:80 --name web1 nginx:alpine
3a7f9c2d1e8b...

22つ目のコンテナを起動

docker run -d -p 8081:80 --name web2 nginx:alpine
9b4c8e1f2a3d...

33つ目のコンテナを起動

docker run -d -p 8082:80 --name web3 nginx:alpine
7e2a9f4c1b6d...

4実行中のコンテナを確認

docker ps
CONTAINER ID IMAGE COMMAND CREATED STATUS PORTS NAMES 7e2a9f4c1b6d nginx:alpine "/docker-entrypoint.…" 10 seconds ago Up 9 seconds 0.0.0.0:8082->80/tcp web3 9b4c8e1f2a3d nginx:alpine "/docker-entrypoint.…" 20 seconds ago Up 19 seconds 0.0.0.0:8081->80/tcp web2 3a7f9c2d1e8b nginx:alpine "/docker-entrypoint.…" 30 seconds ago Up 29 seconds 0.0.0.0:8080->80/tcp web1

全て同じ nginx:alpine イメージから作成されていますが、 独立したコンテナとして動作しています。

5アクセスして確認

ブラウザで以下のURLにアクセスすると、それぞれ独立したNginxサーバーが動作していることを確認できます：

  • http://localhost:8080
  • http://localhost:8081
  • http://localhost:8082

6クリーンアップ

docker stop web1 web2 web3
web1 web2 web3
docker rm web1 web2 web3
web1 web2 web3

📝 書き込み可能レイヤーの仕組み

コンテナを起動すると、イメージの読み取り専用レイヤーの上に、 書き込み可能レイヤーが追加されます。

コンテナのレイヤー構造
書き込み可能レイヤー
コンテナ起動時に作成
変更内容はここに保存
読み取り専用レイヤー
イメージのレイヤー
読み取り専用レイヤー
読み取り専用レイヤー

書き込み可能レイヤーの特徴

  • 一時的: コンテナを削除するとレイヤーも消える
  • 独立: 各コンテナが独自の書き込み可能レイヤーを持つ
  • 変更の保存: ファイル追加、編集、削除はここに記録される
  • 元のイメージは不変: イメージ自体は変更されない

💡 Copy-on-Write（CoW）の仕組み

コンテナ内でファイルを編集すると、以下の流れで処理されます：

  1. イメージレイヤーから該当ファイルをコピー
  2. 書き込み可能レイヤーにコピーを配置
  3. コピーされたファイルを編集

この仕組みにより、イメージを変更せずにファイルを編集できます。

🎯 ハンズオン: コンテナの変更を確認

コンテナ内でファイルを変更して、書き込み可能レイヤーの動作を確認しましょう。

1コンテナを起動

docker run -d -p 8080:80 --name my-web nginx:alpine

2デフォルトのHTMLファイルを確認

docker exec my-web cat /usr/share/nginx/html/index.html
Welcome to nginx! ...

3HTMLファイルを変更

docker exec my-web sh -c 'echo "<h1>Hello Docker!</h1>" > /usr/share/nginx/html/index.html'

ブラウザで http://localhost:8080 にアクセスすると、 「Hello Docker!」と表示されます。

4変更差分を確認

docker diff で、コンテナ内のファイルシステムの変更を確認できます。

docker diff my-web
C /usr C /usr/share C /usr/share/nginx C /usr/share/nginx/html C /usr/share/nginx/html/index.html C /var C /var/cache C /var/cache/nginx A /var/cache/nginx/client_temp A /var/cache/nginx/proxy_temp ...

記号の意味：

  • A: Added（追加されたファイル）
  • C: Changed（変更されたファイル）
  • D: Deleted（削除されたファイル）

5新しいコンテナを起動して確認

同じイメージから新しいコンテナを起動すると、元の状態に戻っていることを確認できます。

docker run -d -p 8081:80 --name my-web2 nginx:alpine
docker exec my-web2 cat /usr/share/nginx/html/index.html
Welcome to nginx! ...

my-web2 は元のファイルを持っています。 これは、イメージ自体は変更されていないためです。

6クリーンアップ

docker rm -f my-web my-web2

💾 コンテナからイメージを作成

コンテナの変更内容を保存して、新しいイメージを作成できます。 これをコミットと呼びます。

docker commit の仕組み
🚢
変更済みコンテナ
書き込み可能レイヤーに変更あり
→
📦
新しいイメージ
変更内容が含まれる

⚠️ docker commit の使いどころ

docker commit は便利ですが、本番環境では推奨されません。

  • 推奨: Dockerfileを使ってイメージを作成（再現性が高い）
  • 非推奨: docker commitで手作業で作成（手順が記録されない）

docker commitは、テストや一時的な変更の保存に使います。

🎯 ハンズオン: コンテナからイメージを作成

カスタマイズしたコンテナから、新しいイメージを作成してみましょう。

1ベースとなるコンテナを起動

docker run -it --name my-custom ubuntu:22.04 bash

2コンテナ内でカスタマイズ

コンテナ内で、パッケージをインストールしたり設定を変更します。

apt update
apt install -y curl vim git
echo "Custom Ubuntu Image" > /root/README.txt
exit

3変更をイメージとして保存

docker commit my-custom my-ubuntu:v1.0
sha256:a1b2c3d4e5f6...

構文: docker commit [コンテナ名] [新しいイメージ名:タグ]

4作成したイメージを確認

docker images
REPOSITORY TAG IMAGE ID CREATED SIZE my-ubuntu v1.0 a1b2c3d4e5f6 5 seconds ago 189MB ubuntu 22.04 fb52e22af1b0 2 weeks ago 77.8MB

my-ubuntu:v1.0 のサイズが元の ubuntu:22.04 より大きくなっています。 これは、インストールしたパッケージが含まれているためです。

5新しいイメージから、コンテナを起動

docker run -it --rm my-ubuntu:v1.0 bash
curl --version
curl 7.81.0 (x86_64-pc-linux-gnu)...
cat /root/README.txt
Custom Ubuntu Image
exit

カスタマイズした内容が保存されていることを確認できました。

6メタデータを付けてコミット

イメージにメッセージや作成者情報を付けることもできます。

docker commit -m "Added curl, vim, git" -a "Your Name" my-custom my-ubuntu:v1.1
docker inspect my-ubuntu:v1.1 | grep -A 5 "Comment"
"Comment": "Added curl, vim, git", "Author": "Your Name",

7クリーンアップ

docker rm my-custom
docker rmi my-ubuntu:v1.0 my-ubuntu:v1.1
