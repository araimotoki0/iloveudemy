🎯 このセクションのゴール

実際のWebアプリケーション開発では、複数のコンテナを連携させて使います。 今回は nginx + Laravel + MySQL の3コンテナ構成を手動で起動してみましょう。

今回作る構成

ブラウザ
→
nginx
:80
→
Laravel
PHP-FPM :9000
→
MySQL
:3306

💡 なぜこの構成？

  • nginx: Webサーバー（リクエストを受け取る）
  • Laravel/PHP-FPM: アプリケーション（PHPを実行）
  • MySQL: データベース（データを保存）

これが実際のWebアプリケーション開発でよく使われる構成です。

🔍 nginxの役割（受付係）

nginxは受付係のような存在です。

  • ブラウザからのリクエストをまず受け取る
  • 静的ファイル（CSS, JS, 画像）→ 自分で直接返す
  • PHPの処理が必要 → PHP-FPMに「これお願い」と転送

PHPだけでもWebサーバーは動きますが、nginxを前に置くことで高速化・負荷分散ができます。
※ この「間に入って振り分ける」役割をリバースプロキシと呼びます。

🐘 PHP-FPMとは？

PHPのコードを実行する専門スタッフのようなものです。

🍔 ハンバーガー屋さんで例えると...

昔のやり方（CGI）

お客さんが来るたびに → 料理人を雇う → 料理を作る → 料理人を解雇
→ 次のお客さんが来たら → また料理人を雇う...

😰 毎回雇うの遅すぎ！

※ CGI = Common Gateway Interface（1990年代の技術）
リクエストごとに php index.php を実行→終了→また実行...

今のやり方（PHP-FPM）

料理人を最初から何人か雇っておく → お客さんが来たらすぐ作れる
→ 作り終わったら次の注文を待つ → ずっと待機

🚀 すぐ対応できて速い！

※ PHPプロセスが常に起動した状態でリクエストを待ち受ける

FPM = FastCGI Process Manager
「高速に処理するためのプロセス（料理人）を管理する仕組み」という意味です。

🌐 他の言語にもあるの？

はい！言語ごとに名前は違いますが、同じ役割のものがあります。

PHP PHP-FPM
Python Gunicorn, uWSGI
Ruby Puma, Unicorn
Node.js 自身がサーバーになる（別途不要）

「nginx（受付） + アプリサーバー（料理人）」の構成は、どの言語でも共通のパターンです。

📁 作業ディレクトリの準備

1プロジェクトフォルダを作成

mkdir -p ~/docker-laravel
cd ~/docker-laravel

2必要なフォルダを作成

# nginx設定ファイル用
mkdir -p docker/nginx

3nginx設定ファイルを作成

エディタで docker/nginx/default.conf を作成し、以下の内容を貼り付けてください。

docker/nginx/default.conf
server {
    listen 80;
    server_name localhost;
    root /var/www/html/public;
    index index.php index.html;

    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    location ~ \.php$ {
        fastcgi_pass app:9000;
        fastcgi_index index.php;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
        include fastcgi_params;
    }
}

📝 設定の説明

基本設定

listen 80 ポート80（HTTP）でリクエストを待ち受ける
server_name localhost このサーバーが応答するホスト名（開発環境なのでlocalhost）
root /var/www/html/public ドキュメントルート（Laravelはpublicフォルダが公開対象）
index index.php index.html ディレクトリにアクセスした時に探すファイル（優先順）

リクエストの振り分け

location / すべてのリクエストに対するルール
location ~ \.php$ .phpで終わるURLへのリクエストを処理（~は正規表現）

🔍 try_files の動き

try_files $uri $uri/ /index.php?$query_string

「このURLにアクセスが来たら、順番に試してみて」という意味です。

例: /users/123 にアクセスが来た場合

① $uri /users/123 というファイルがある？ → ❌ ない
② $uri/ /users/123/ というフォルダがある？ → ❌ ない
③ /index.php → ✅ index.php に転送！Laravelが処理

例: /css/app.css にアクセスが来た場合

① $uri /css/app.css というファイルがある？ → ✅ ある！そのまま返す

💡 CSSや画像など実ファイルはそのまま返し、
/users/123 のようなURLはLaravelに任せる、という振り分けです。

「try_filesでindex.phpに行ったら、その後どうなるの？」

① /users/123 にアクセス

↓

② location / がマッチ

↓

③ try_files で /index.php に書き換え

↓

④ nginx が再度どのlocationか評価 ← ポイント！

↓

⑤ location ~ \.php$ がマッチ

↓

⑥ PHP-FPMに転送 → Laravelが処理

try_filesで書き換わった後、nginxはもう一度locationを評価し直します。

PHP-FPMへの転送設定

① location ~ \.php$

「URLが .php で終わるリクエストを処理」

location 「このURLパターンの時は...」
~ 正規表現を使う
\.php$ .php で終わる（$は「末尾」の意味）

例: /index.php ✅、/admin/login.php ✅、/style.css ❌

② fastcgi_pass app:9000

「PHPの処理をappコンテナに投げて」

app コンテナ名（Dockerネットワークで名前解決）
9000 PHP-FPMが待ち受けているポート

③ fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name

環境変数をセットしてPHP-FPMに渡す

fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name
　　　　　　　　↑ 変数名　　　　　　　↑ セットする値
$document_root /var/www/html/public（rootで設定した値）
$fastcgi_script_name /index.php（リクエストされたファイル）
合体すると /var/www/html/public/index.php

PHP側では $_SERVER['SCRIPT_FILENAME'] でアクセスできる

④ include fastcgi_params

その他の情報もPHP-FPMに渡す：

  • REQUEST_METHOD（GET? POST?）
  • QUERY_STRING（?id=123 の部分）
  • REMOTE_ADDR（アクセス元のIP）など

PHPで $_GET, $_POST, $_SERVER が使えるのはこれのおかげ

🌐 Dockerネットワークの作成

複数のコンテナがお互いに通信するために、専用のネットワークを作成します。

4ネットワークを作成

docker network create my-laravel-nw
abc123def456...

💡 なぜネットワークが必要？

同じネットワークに所属するコンテナは、コンテナ名で通信できます。

  • nginx → app:9000 でLaravelコンテナに接続
  • Laravel → db:3306 でMySQLコンテナに接続

コンテナのIPアドレスは再起動のたびに変わる可能性があります。
IPアドレスではなくコンテナ名で指定すれば、IPが変わっても接続できます。
※ ネットワークの詳細は後のセクションで学びます。

🗄️ MySQLコンテナを起動

まずデータベースから起動します。他のコンテナがDBに依存するためです。

5MySQL設定ファイルを作成

文字コードとタイムゾーンを設定します。エディタで docker/mysql/my.cnf を作成してください。

mkdir -p docker/mysql
docker/mysql/my.cnf
[mysqld]
character-set-server=utf8mb4
collation-server=utf8mb4_unicode_ci
default-time-zone='Asia/Tokyo'

[client]
default-character-set=utf8mb4

📝 設定の説明

character-set-server 文字コード（utf8mb4で絵文字も対応）
collation-server 照合順序（文字の比較ルール）
default-time-zone タイムゾーン（日本時間）

6MySQLコンテナを起動

docker container run -d \ --name db \ --network my-laravel-nw \ -e MYSQL_ROOT_PASSWORD=secret \ -e MYSQL_DATABASE=laravel \ -e MYSQL_USER=laravel \ -e MYSQL_PASSWORD=secret \ -v mysql-data:/var/lib/mysql \ -v $(pwd)/docker/mysql/my.cnf:/etc/mysql/conf.d/my.cnf \ mysql:8.0

長い... でもこれでやっと1つ目のコンテナが起動しました。

💡 なぜ mysql:8.0 を選んだ？

MySQLは公式イメージのみ提供されており、slimやalpine版はありません。
バージョン8.0は安定版で、Laravel等のフレームワークとの互換性が高いです。

📦 -v の2つの書き方

このコマンドでは -v を2回使っていますが、書き方が違います：

ボリューム mysql-data:/var/lib/mysql Dockerが管理する領域にデータを保存
バインドマウント $(pwd)/docker/mysql/my.cnf:... ホストのファイルをコンテナにマウント

見分け方： / ./ $(pwd) で始まる → バインドマウント、名前だけ → ボリューム
※ 「runオプション」セクションでも少し説明しています。ボリュームの詳細は後のセクションで学びます。

📝 オプションの説明

--name db コンテナ名を「db」に設定
--network 作成したネットワークに参加
-e MYSQL_... 環境変数でDB設定
-v mysql-data:... データを永続化

🐘 Laravel/PHPコンテナを起動

次にアプリケーションコンテナを起動します。

7Laravelプロジェクトを作成

まずComposerでLaravelプロジェクトを作成します。

docker container run --rm \ -v $(pwd):/app \ -w /app \ composer:latest \ composer create-project laravel/laravel src

これで src/ フォルダにLaravelがインストールされます（少し時間がかかります）。

💡 なぜ先にLaravelを作成する？

PHPコンテナ起動時に -v $(pwd)/src:/var/www/html でマウントするため、
先に src/ フォルダ（Laravelのコード）が存在している必要があります。

空のフォルダをマウントすると、コンテナ内も空になってしまいます。

8PHP設定ファイルを作成

エディタで docker/php/php.ini を作成してください。

mkdir -p docker/php
docker/php/php.ini
[PHP]
; タイムゾーン
date.timezone = Asia/Tokyo

; メモリ制限
memory_limit = 256M

; ファイルアップロード
upload_max_filesize = 32M
post_max_size = 32M

; エラー表示（開発環境用）
display_errors = On
error_reporting = E_ALL

9PHP用のDockerfileを作成

php:8.4-fpm イメージにはLaravelに必要な拡張が含まれていません。
エディタで docker/php/Dockerfile を作成してください。

docker/php/Dockerfile
FROM php:8.4-fpm

# 必要なパッケージをインストール
RUN apt-get update && apt-get install -y \
    libzip-dev \
    unzip \
    && rm -rf /var/lib/apt/lists/*

# PHP拡張をインストール
RUN docker-php-ext-install \
    pdo_mysql \
    zip \
    bcmath

# php.ini をコピー
COPY php.ini /usr/local/etc/php/conf.d/php.ini

# 作業ディレクトリ
WORKDIR /var/www/html

📝 Dockerfileの説明

apt-get install PHP拡張に必要なライブラリをインストール
pdo_mysql MySQLに接続するための拡張
zip Composerやファイル操作に必要
bcmath 高精度の計算に使用（Laravelで推奨）
COPY php.ini 設定ファイルをコンテナにコピー

💡 なぜ php:8.4-fpm を選んだ？

alpine版（php:8.4-fpm-alpine）もありますが、apt-get が使えません（alpineは apk）。
拡張機能のインストールでエラーが起きやすく、学習段階では標準イメージがおすすめです。

10PHPイメージをビルド

docker image build -t my-php:8.4-fpm docker/php

ポイント: -t my-php:8.4-fpm でイメージに名前を付けています。

11PHP-FPMコンテナを起動

docker container run -d \ --name app \ --network my-laravel-nw \ -v $(pwd)/src:/var/www/html \ -e DB_CONNECTION=mysql \ -e DB_HOST=db \ -e DB_PORT=3306 \ -e DB_DATABASE=laravel \ -e DB_USERNAME=laravel \ -e DB_PASSWORD=secret \ my-php:8.4-fpm

また長い... 2つ目のコンテナが起動しました。

12データベースのマイグレーション

Laravelが使用するテーブルを作成します。

docker container exec app php artisan migrate

「Yes」を入力してマイグレーションを実行してください。

🌐 nginxコンテナを起動

最後にWebサーバーを起動します。

13nginxコンテナを起動

docker container run -d \ --name web \ --network my-laravel-nw \ -p 8080:80 \ -v $(pwd)/src:/var/www/html \ -v $(pwd)/docker/nginx/default.conf:/etc/nginx/conf.d/default.conf \ nginx:1.28-alpine

💡 なぜnginxにもソースコードをマウント？

appコンテナ（PHP-FPM）と同じ -v $(pwd)/src:/var/www/html がありますね。
実は両方のコンテナに必要です。リクエストの種類によって処理が違うからです。

リクエスト 処理の流れ 必要なコンテナ
CSS/JS/画像 nginx → 直接ファイルを返す nginx
PHPページ nginx → PHP-FPM → 実行結果を返す PHP-FPM
📁 ホストPC
└── src/public/
index.php ← PHP-FPMが読んで実行
css/app.css ← nginxが直接返す
js/app.js ← nginxが直接返す
↓ 同じファイルを両方にマウント ↓
🌐 nginx
/var/www/html
静的ファイル配信
🐘 PHP-FPM
/var/www/html
PHP実行

⚠️ nginxにマウントがないと → CSS/JS/画像が全て404エラー
⚠️ PHP-FPMにマウントがないと → PHPファイルが見つからずエラー

💡 なぜ nginx:1.28-alpine を選んだ？

nginxは設定ファイルを読み込むだけなので、alpine版でも問題なく動作します。
イメージサイズが小さく（約40MB vs 190MB）、本番環境でもよく使われます。

😱 問題1: コマンドが長すぎる

MySQLコンテナを起動するコマンドだけでこれだけ...

docker container run -d --name db --network my-laravel-nw -e MYSQL_ROOT_PASSWORD=secret -e MYSQL_DATABASE=laravel -e MYSQL_USER=laravel -e MYSQL_PASSWORD=secret -v mysql-data:/var/lib/mysql mysql:8.0

😱 問題2: 起動順序を間違えると動かない

  • 1. ネットワーク作成
  • 2. MySQL起動（先に起動しないとLaravelが接続できない）
  • 3. Laravel起動
  • 4. nginx起動

順番を間違えたら？ → エラーになる

😱 問題3: 設定がどこにも残らない

  • コマンド履歴から探す？
  • メモ帳に書いておく？
  • チームメンバーにどう共有する？

😱 問題4: 停止・削除も面倒

# 停止（3回）
docker container stop web app db
# 削除（3回）
docker container rm web app db
# ネットワーク削除
docker network rm my-laravel-nw

💡 解決策: Docker Compose

これらの問題をすべて解決するのがDocker Composeです！

Docker Composeでできること

  • ✅ 設定をファイル（yml）に保存
  • ✅ 1コマンドで全コンテナ起動
  • ✅ 起動順序を自動管理
  • ✅ 1コマンドで全コンテナ停止・削除
  • ✅ チームで設定を共有できる
