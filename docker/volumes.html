💾 ボリュームとは

Docker Compose で mysql-data:/var/lib/mysql と書きましたよね？
なぜこれが必要なのか、体験しながら理解しましょう。

覚えることは2つだけ
1
名前付きボリューム

データを永続化する

2
バインドマウント

ソースコードを同期する

実際に手を動かして体験していきましょう。

❌ ハンズオン1: データが消える

コンテナを削除すると、中のデータはどうなるでしょう？
実際に確認してみましょう。

Step 1: コンテナ内にファイルを作成
# コンテナを起動
docker container run -d --name test alpine sleep 3600
# コンテナ内にファイルを作成
docker container exec test sh -c "echo 'Hello Docker' > /data.txt"
# ファイルを確認
docker container exec test cat /data.txt
Hello Docker

コンテナ内に /data.txt を作成しました。

Step 2: コンテナを削除して再作成
# コンテナを削除
docker container rm -f test
# 新しいコンテナを起動
docker container run -d --name test alpine sleep 3600
# ファイルを確認
docker container exec test cat /data.txt
cat: can't open '/data.txt': No such file or directory
結果: データが消えた！

コンテナを削除すると、中のデータも一緒に消えます。

なぜ消える？

コンテナは「使い捨て」が基本思想です。

コンテナ
data.txt
→ 削除 →
消滅
data.txt

DBのデータも同じ。docker compose down すると消えます。
だからボリュームが必要なんです。

Step 3: 後片付け
docker container rm -f test

✅ ハンズオン2: ボリュームで永続化

名前付きボリュームを使うと、コンテナを消してもデータが残ります。

Step 1: ボリュームを使ってファイルを作成
# mydata ボリュームを /data にマウントして起動
docker container run -d --name test -v mydata:/data alpine sleep 3600
# ボリューム内にファイルを作成
docker container exec test sh -c "echo 'Hello Docker' > /data/test.txt"
# ファイルを確認
docker container exec test cat /data/test.txt
Hello Docker

-v mydata:/data で名前付きボリューム「mydata」を /data にマウントしました。

💡 名前付きボリュームの書き方

見分け方：パスで始まるか、名前だけか

mydata:/data
↑ 名前だけ = 名前付きボリューム
./src:/var/www/html
↑ ./ で始まる = バインドマウント

マウント単位：

  • • 名前付きボリューム → ディレクトリ単位のみ
  • • バインドマウント → ディレクトリもファイルもOK
Step 2: コンテナを削除して再作成
# コンテナを削除
docker container rm -f test
# 新しいコンテナで同じボリュームをマウント
docker container run -d --name test -v mydata:/data alpine sleep 3600
# ファイルを確認
docker container exec test cat /data/test.txt
Hello Docker
結果: データが残ってる！

コンテナを削除しても、ボリュームのデータは消えません。

なぜ残る？
コンテナ
/data → mydata
→ 削除 →
消滅
mydata ボリューム
test.txt → 残ってる！

ボリュームはコンテナとは別に管理されるので、消えません。

Step 3: 後片付け
docker container rm -f test
# ボリュームも削除する場合
docker volume rm mydata
ボリュームはどこにある？
# ボリューム一覧を確認
docker volume ls
DRIVER VOLUME NAME local mydata local mysql-data

名前付きボリュームは Docker が管理する場所に保存されます。
ホストから直接見ることはできません（見る必要もありません）。

保存場所: /var/lib/docker/volumes/mydata/_data
→ Docker に任せておけばOK

📂 ハンズオン3: バインドマウント

バインドマウントは、ホスト（自分のPC）のフォルダをコンテナに接続します。
開発時にソースコードを同期するのに使います。

Step 1: ホストにファイルを作成
# テスト用フォルダを作成
mkdir -p ~/docker-test && cd ~/docker-test
# HTMLファイルを作成
echo '<h1>Hello from Host!</h1>' > index.html
Step 2: バインドマウントでコンテナ起動
# カレントディレクトリをnginxの公開フォルダにマウント
docker container run -d --name web -p 8080:80 -v $(pwd):/usr/share/nginx/html nginx

-v $(pwd):/usr/share/nginx/html で現在のフォルダをマウントしました。

# ブラウザで確認
open http://localhost:8080 # Mac
# または http://localhost:8080 をブラウザで開く
Step 3: ホスト側でファイルを編集
# ホスト側でファイルを変更
echo '<h1>Updated!</h1>' > index.html
結果: ブラウザをリロード → 即座に反映！

コンテナを再起動しなくても、ホスト側の変更がリアルタイムで反映されます。

これが開発で便利な理由
ホスト（あなたのPC）
VSCode で編集
⟷
コンテナ
即座に反映

ソースコードをホストで編集 → コンテナに自動反映 → ブラウザで確認

双方向に同期される

バインドマウントは双方向です。コンテナ内で作ったファイルもホストに反映されます。

# コンテナ内でファイルを作成
docker container exec web touch /usr/share/nginx/html/new-file.txt
# → ホスト側にも new-file.txt が出現！

Laravel の composer install で vendor/ が作られるのも同じ仕組みです。

Step 4: 後片付け
docker container rm -f web
rm -rf ~/docker-test

🐳 Docker Compose での書き方

Docker Compose では volumes: で簡単に設定できます。

よくある構成
# docker-compose.yml
services:
  app:
    build: .
    volumes:
      - ./src:/var/www/html # バインドマウント（ソースコード）
  db:
    image: mysql:8.0
    volumes:
      - mysql-data:/var/lib/mysql # 名前付きボリューム（DBデータ）
volumes:
  mysql-data: # ボリュームを定義
→ ./src:/var/www/html = バインドマウント（./ で始まる）
→ mysql-data:/var/lib/mysql = 名前付きボリューム（名前だけ）
見分け方
バインドマウント
./src:/var/www/html

パス（./ や /）で始まる

名前付きボリューム
mysql-data:/var/lib/mysql

名前だけ（トップレベルで定義）

注意: docker compose down -v
# 通常の停止（ボリュームは残る）
docker compose down
# ボリュームも削除（データが消える！）
docker compose down -v

-v をつけると名前付きボリュームも削除されます。
DBのデータを消したくない場合は -v をつけないでください。

なぜDBにバインドマウントを使わない？

「ソースコードと同じように ./mysql-data:/var/lib/mysql でいいのでは？」と思うかもしれませんが...

❌ バインドマウント
  • • rm -rf で間違って消すリスク
  • • Git に間違えてコミットするリスク
✅ 名前付きボリューム
  • • Docker が安全に管理
  • • 間違って消すリスクが低い
  • • Git の管理対象外

🎯 まとめ

ボリュームの本質
!
コンテナを消すとデータも消える

コンテナは使い捨て。中のデータは永続化されない。

1
名前付きボリューム = データ永続化

DBのデータなど、消えてほしくないものに使う。

mysql-data:/var/lib/mysql
2
バインドマウント = ソースコード同期

ホストで編集 → コンテナに即反映。開発時に使う。

./src:/var/www/html
使い分け
用途 方式 例
DBデータ 名前付きボリューム mysql-data:/var/lib/mysql
ソースコード バインドマウント ./src:/var/www/html
設定ファイル バインドマウント ./php.ini:/usr/local/etc/php/php.ini

✅ 理解できたこと

  • コンテナを消すとデータも消える
  • 名前付きボリュームでデータを永続化できる
  • バインドマウントでソースコードを同期できる
  • docker compose down -v はボリュームも消える
